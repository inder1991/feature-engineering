# SP-1.5 — Metadata Overlay Hardening (Design Spec)

**Status:** Designed / agreed in dialogue (2026-07-03). Not yet planned or built.
**Relationship:** Sibling of SP-0.5 (backbone hardening, merged `631bcb7`). Hardens the built SP-1 metadata overlay's **read path** so it is correct and operable **before SP-3 (grounding) freezes the overlay's seams**. Scope companion: the deferred human-ops/scale items become **SP-1.6**.
**Source:** the 2026-07-02 platform review (`2026-07-02-platform-review-sp0-sp15.md` §5, findings #2/#3/#7/#9/#10 + the #1 residual), re-grounded in the current code 2026-07-03.

---

## 1. Purpose and scope

SP-1's write side is strong (event-sourced fact aggregate, CAS confirmation races, dual-owner joins). But the review found the overlay "a correct kernel mistaken for a working system": in one place its read path **fails open** (the exact leakage vector it exists to block), its freshness model **guarantees recurring outages**, and parts of it cannot be operated against a real estate. SP-0.5 already closed the worst *operational* gap — its daemon now drives the `OverlayProjection` read model and the expiry poller every tick — so SP-1.5 is narrower than the review first framed it.

SP-1.5 is a **tight read-path hardening pass**. It fixes the correctness/safety items that SP-3 would otherwise freeze into its assumptions, plus the one persistence-seam fix (multi-catalog keying) that is cheap now and expensive later.

### 1.1 In scope (six fixes)

1. Read-time expiry enforcement in `resolve_fact` (close the fail-open).
2. Multi-catalog keying of the overlay's secondary indexes (the one seam-fix pulled forward).
3. Pre-expiry renewal + configurable per-fact-type TTL (end the guaranteed outage windows).
4. A production driver for catalog-drift detection (finish the SP-0.5 "no drivers" residual).
5. STALE-confirm dangling-referent validation (stop re-blessing renamed/dropped columns).
6. Server-side profiler schema allowlist (close the self-attesting fail-open).

### 1.2 Non-goals (deferred to SP-1.6)

Group ownership + delegation; bulk confirmation ergonomics; the full multi-catalog **adapter registry** (SP-1.5 does the *keying*, not the registry); governance-queue repair/re-route; fact-vocabulary expansion (data-class / sensitivity / retention facts). These are real but they change how humans *confirm* facts, not the read seam SP-3 builds against, and each is closer to a feature than a hardening fix.

### 1.3 Why "before SP-3"

SP-3 consumes the overlay through the **read seam** — `resolve_fact` and the `fact_key` shape — and does not care how a fact was confirmed. So the items that would get frozen wrong into SP-3 are the read-path ones (expiry enforcement, the fact-serving contract) and the persistence keys (`fact_key` / `ref_object`). Those are exactly fixes 1 and 2, which are sequenced first.

### 1.4 Execution model

Same as SP-0.5: subagent-driven development, TDD per task, adversarial review per task, keep the full suite green (currently 1080 passed). Six tasks, ordered §8.

## 2. Fix 1 — Read-time expiry enforcement (finding #3, the fail-open)

**Problem.** `resolve_fact` (`src/featuregen/overlay/resolve.py:162-226`) reads `overlay_fact_state`; its `SELECT` already fetches `expires_at` (line ~193), but the VERIFIED branch (`if status == "VERIFIED": return _overlay_verified(...)`, ~216-219) returns the value with **zero reference to `expires_at`**. The persisted status only flips away from VERIFIED asynchronously (the `overlay_expiry` timer fires → `OVERLAY_FACT_EXPIRED` appended → `OverlayProjection.apply` flips status→REVERIFY). Between `expires_at` passing and that async catch-up, `resolve_fact` serves a value past its expiry — the one component whose contract is fail-closed, failing open. For `availability_time` facts this is the point-in-time leakage the overlay exists to prevent.

**Fix.** In the VERIFIED branch, compare `expires_at` to the clock and fail closed when past:

- Add a reason constant `_REASON_EXPIRED = "expired_pending_reverify"` beside the existing reason constants.
- Change the VERIFIED branch to: if `row["expires_at"] is not None and row["expires_at"] <= now` → return `_overlay_blocked("REVERIFY", _REASON_EXPIRED, obj, fact_type, use_case, prior_value=row["value"])` (surfacing the old value as read-only context, exactly as the projection will once `EXPIRED` applies); otherwise return `_overlay_verified(...)` unchanged.
- Reuse the existing `_overlay_blocked` constructor and the `REVERIFY` status (already in `FactStatus`). `expires_at` is `timestamptz` (tz-aware from psycopg); `NULL` means "no TTL" and stays servable.

This closes the **time-expiry** vector regardless of poller/projection lag, because `expires_at` is an absolute instant already in the row — the fix does not depend on the daemon having caught up.

**Clock / testability.** Add `now: datetime | None = None` as an optional trailing keyword to `resolve_fact`, defaulting to `datetime.now(UTC)` (already imported). Tests pass a fixed `now` to make before/after-expiry deterministic; this also keeps `resolve_fact`'s expiry threshold on the same Python-clock basis the expiry poller already uses (`worker.py` threads `now=datetime.now(UTC)` into `fire_due_overlay_expiries`). Do the comparison in Python, not SQL — a SQL `now()` would introduce a second clock source that could disagree with the poller by skew.

**Seam impact (SP-3).** A backward-compatible optional `now` kwarg. SP-3 must treat "status REVERIFY carrying a `prior_value`" as **blocked, not usable** — which the fail-closed contract already requires (only VERIFIED is servable). Returning status VERIFIED with `value=None` was rejected precisely because it would break a naive `if status=='VERIFIED': use(value)` consumer.

**Deferred (tracked follow-up, not in this fix).** The **catalog-driven STALE lag** vector is *not* time-based — a STALE fact keeps a future `expires_at`, so the read-time check cannot cover it. Closing it would require, per read, checking whether the fact has an `OVERLAY_FACT_STALED/EXPIRED/REJECTED` event beyond the overlay checkpoint (`read_as_of(conn, "overlay")`). Because SP-0.5's daemon now bounds projection lag to ~1–2 ticks (~seconds), this residual is **deferred to a tracked follow-up** rather than adding an events-table query to every resolve. Revisit if the lag window proves material.

**Tests.** A VERIFIED row with `expires_at` in the past → resolve returns blocked/`expired_pending_reverify` with the value surfaced as `prior_value`, not usable. A VERIFIED row with `expires_at` in the future (or NULL) → served normally. Both driven with an injected `now`.

## 3. Fix 2 — Multi-catalog keying (finding #10, the seam-fix)

**Problem.** `fact_key` embeds `catalog_source` (`src/featuregen/overlay/identity.py:65-89`), but `display_object_ref` **omits it** (`identity.py:92-99`), and that catalog-source-less string is the key for: the dependency index (`overlay_fact_dependency.ref_object`, written by `projection.py`), the drift fingerprint snapshot (`overlay_catalog_object.object_ref`, `catalog_changes.py`), and the fixture catalog maps. The moment a second catalog is onboarded (the explicit next step per the deferral), two `core.transactions` tables from different sources **collide** — drift in catalog A stales facts about catalog B, and the dependency index cross-fires. The one thing `fact_key` was designed to prevent re-enters through every secondary index.

**Fix (keying only — not the adapter registry).** Key `overlay_fact_dependency.ref_object` and the fingerprint snapshot `overlay_catalog_object.object_ref` on `(catalog_source, object_ref)` — do this migration now, while the tables are small and rebuildable (the projection read models are droppable and deterministically rebuildable). Update the projection writer and the drift snapshot/anti-join to carry `catalog_source`. **Do not** build the multi-catalog adapter registry or change the process-global single adapter (`catalog.py:223-243`) — that is SP-1.6.

**Seam impact (SP-3).** This is the load-bearing seam change: it guarantees SP-3 grounds against **catalog-source-qualified** references rather than freezing a single-catalog assumption. `fact_key` already qualifies; this aligns the secondary indexes and the referent strings so the whole read/dependency surface is source-qualified.

**Coupling.** Fix 5 (STALE-confirm referents) validates `ref_object` strings against the adapter — so it must land *after* this keying change to validate against source-qualified referents. Sequenced accordingly (§8).

**Tests.** Two facts on same-named objects in different `catalog_source`s do not collide in the dependency index or the fingerprint snapshot; drift in one catalog does not stale the other's facts. Projection rebuild reproduces the source-qualified keys.

## 4. Fix 3 — Pre-expiry renewal + configurable TTL (finding #2)

**Problem.** Re-verification can only begin **after** a fact has expired: `_AWAITING_CONFIRMATION` (`src/featuregen/overlay/_lifecycle.py`) excludes VERIFIED, so a still-VERIFIED fact cannot be re-confirmed until the `overlay_expiry` timer flips it to REVERIFY — i.e. until dependent pipelines are already blocked. Combined with the hardcoded, un-jittered `_DEFAULT_TTL = 180 days`, an onboarding wave of facts all expire the same day, stalling whole domains simultaneously. Every confirmed fact takes a mandatory recurring outage.

**Fix.**
- **Pre-expiry renewal window.** Arm a "renewal reminder" timer alongside the `overlay_expiry` timer at `expires_at − grace` (grace configurable, e.g. 14 days). When it fires, open the re-verify task **while the fact stays VERIFIED and servable until the real `expires_at`**. A data owner re-confirming within the grace window renews the fact **without a VERIFIED→DRAFT regression** — the fold's `_NON_TERMINAL` rule (no VERIFIED→DRAFT) is preserved; renewal replaces the VERIFIED value in place with a fresh `expires_at`, it does not drop to DRAFT.
- **Configurable per-fact-type/risk-tier TTL + jitter.** Replace the single hardcoded 180-day horizon with a horizon resolved per fact type (and, where applicable, per risk tier), sealed as server-side config at `register_overlay`/deploy time, plus jitter so an onboarding wave does not expire in lockstep.
- **Confirmable-within-grace.** Extend the confirm path so a VERIFIED fact with an **open renewal task** (inside the grace window) is confirmable; a VERIFIED fact with no open renewal task remains non-confirmable (unchanged — no spurious re-confirmations).

**Seam impact (SP-3).** None. The fact stays VERIFIED and servable throughout the grace window, so SP-3 observes no new read state; only the human/renewal workflow changes.

**Tests.** A fact within its grace window is still served VERIFIED *and* has an open re-verify task; re-confirming it inside the window renews `expires_at` with no DRAFT regression and no read-path interruption. TTL is resolved per fact type from config (not the constant) and jittered. A fact past `expires_at` with no renewal is blocked (Fix 1) — the two fixes compose: renewal *prevents* reaching that state on the happy path.

## 5. Fix 4 — Catalog-drift detection driver (finding #1 residual)

**Problem.** SP-0.5's daemon (`src/featuregen/runtime/worker.py`) drives the `OverlayProjection` read model and runs `fire_due_overlay_expiries` every tick — but it does **not** drive `detect_catalog_changes` (`src/featuregen/overlay/catalog_changes.py`). So in production, schema drift never stales dependent facts: a dropped/renamed column never flips its dependents to STALE. The mechanism exists and is tested, but nothing runs it. (This is why Fix 5 matters — without a live driver, STALE is never reached at all.)

**Fix.**
- **Wire it as a worker stage** analogous to `_advance_overlay_expiries`: skip cleanly when no catalog adapter is wired; run inside its own transaction; count/log via the observability counters.
- **On a slower cadence than the 1-second tick.** `detect_catalog_changes` pages the whole catalog, so it must not run every tick — run it on a configurable interval (e.g. every N ticks / minutes), tracked by a last-run marker, not on every `run_worker_once`.
- **Minimum scale fixes to make it safe to run** (from finding #11): batch the snapshot writes (COPY / values-list instead of one round-trip per object), chunk the staling + re-verify-task opening into separate transactions instead of one giant lock-holding transaction, and fix the **dead `columns_fingerprint`** (currently written as `""`, so column-level drift is never detected).

**Seam impact (SP-3).** None on the read path; it keeps STALE detection alive so the fail-closed floor actually engages on drift.

**Tests.** A simulated catalog change (dropped/renamed column) run through the driver stales the dependent facts and opens re-verify tasks; column-level drift is detected (the fingerprint is no longer dead); the driver honors its slower cadence and skips when no adapter is wired.

## 6. Fix 5 — STALE-confirm dangling-referent validation (finding #9)

**Problem.** On confirm of a REVERIFY/STALE fact, the default confirmed value is `state.prior_value`, validated **only against the JSON schema, never the live catalog** (`src/featuregen/overlay/confirmation_commands.py:~96-106`); `catalog_changes.py` then advances the snapshot so the dangling name never re-triggers detection. Result: a human clicking "confirm" on a rename-staled `availability_time` fact yields a VERIFIED fact pointing at a **nonexistent column** until its next TTL expiry — drift detection becomes drift laundering, and downstream grounding emits SQL against columns that no longer exist.

**Fix.** On confirm of a STALE/REVERIFY fact, extract every referenced object (the same `_dependencies`/referent extraction the projection uses) and validate each against the **live adapter**. If any referent is gone (renamed/dropped), reject the default `prior_value` re-confirm and require an **explicit value override** naming the current object(s). A clean re-confirm (all referents still present) proceeds as today.

**Seam impact (SP-3).** None on reads; it tightens the confirm path so a VERIFIED fact's referents are guaranteed to exist at confirm time. Lands **after Fix 2** because referents are `ref_object` strings that Fix 2 makes catalog-source-qualified.

**Tests.** Confirming a STALE fact whose referent column was renamed is rejected unless an override is supplied; confirming a STALE fact whose referents all still exist succeeds; the override path yields a VERIFIED fact pointing at live objects.

## 7. Fix 6 — Server-side profiler allowlist (finding #7, fail-open)

**Problem.** The profiler's schema allowlist is caller-supplied: `allowed_schemas = frozenset(cmd.args.get("allowed_schemas", ()))` (`src/featuregen/overlay/profiler_command.py:~55`), and the check tests the target against a list **the same command sent in**. Any principal holding `run_profiler` can profile HR/payroll/general-ledger by naming it in their own `allowed_schemas`. The security-audit denial only fires for targets outside the caller's own list — a control the accessor defines is not a control. (The read-only guarantee is also only *simulated* via `SET LOCAL transaction_read_only` on the full-privilege platform connection.)

**Fix.** Move the allowlist to **server-side config sealed at `register_overlay`/deploy time** — the profiler resolves the permitted schemas from that config, never from `cmd.args`. A `run_profiler` for a schema outside the sealed allowlist is denied (and audited) regardless of what the caller passes. Expose the allowlist as a config surface set at registration.

**Optional within this fix (state explicitly in the plan):** run the profiler scan over a genuinely restricted read-only role/connection rather than the `SET LOCAL` simulation on the full-privilege connection. If standing up a restricted role is out of scope for the daemon/deployment in this pass, keep the `SET LOCAL` simulation but document it as a deployment-hardening follow-up — the load-bearing fix here is removing the self-attestation.

**Seam impact (SP-3).** None.

**Tests.** A `run_profiler` naming a schema outside the sealed server-side allowlist is denied + audited even though the caller passed it in `allowed_schemas`; a scan of an allowed schema proceeds.

## 8. Task ordering and dependencies

Six tasks, ordered so the read-seam fixes land first and the dependent fixes follow their prerequisites:

1. **Read-time expiry enforcement** (Fix 1) — read seam; standalone.
2. **Multi-catalog keying** (Fix 2) — read/persistence seam; must precede Fix 5.
3. **Pre-expiry renewal + configurable TTL** (Fix 3) — freshness workflow; standalone.
4. **Catalog-drift detection driver** (Fix 4) — worker stage + scale fixes; standalone.
5. **STALE-confirm dangling-referent validation** (Fix 5) — depends on Fix 2 (source-qualified referents).
6. **Server-side profiler allowlist** (Fix 6) — standalone.

Fixes 1 and 2 are the SP-3-blocking ones; if SP-1.5 must be split, that is the minimum viable slice.
