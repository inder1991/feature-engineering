# SP-1.5 — Metadata Overlay Hardening (Design Spec)

**Status:** Designed / agreed in dialogue (2026-07-03); **review round 1 incorporated** (16 findings → §13). Not yet planned or built.
**Relationship:** Sibling of SP-0.5 (backbone hardening, merged `631bcb7`). Hardens the built SP-1 metadata overlay's **read path** so it is correct and operable **before SP-3 (grounding) freezes the overlay's seams**. Deferred human-ops/scale items become **SP-1.6**.
**Source:** the 2026-07-02 platform review (`2026-07-02-platform-review-sp0-sp15.md` §5) + review round 1 (2026-07-03), re-grounded in the current code.

---

## 1. Purpose and scope

SP-1's write side is strong (event-sourced fact aggregate, CAS confirmation races, dual-owner joins). But the review found the overlay "a correct kernel mistaken for a working system": in places its read path **fails open** (the exact leakage vector it exists to block), its freshness model **guarantees recurring outages**, and parts cannot be operated against a real estate. SP-0.5 already closed the worst *operational* gap — its daemon now drives the `OverlayProjection` read model and the expiry poller every tick — so SP-1.5 is a **tight read-path hardening pass**: fix the correctness/safety items SP-3 would otherwise freeze, plus the one persistence-seam fix (multi-catalog keying) that is cheap now and expensive later.

### 1.1 In scope

- **Foundations** (§3): `OverlayConfig` (sealed server-side config) and a **canonical source-qualified catalog-reference model** across the whole read surface.
- **Fix 1** (§4): read-time **freshness guard** in `resolve_fact` — fail closed on both time-expiry *and* stale drift.
- **Fix 2** (§5): the canonical source-qualified reference model applied to every read-model sink, with an explicit migration/replay.
- **Fix 3** (§6): pre-expiry renewal + configurable per-fact-type TTL, with fully-specified timer + confirm-in-grace mechanics, covering single facts *and* joins.
- **Fix 4** (§7): a **crash-safe, multi-worker-safe** production driver for catalog-drift detection.
- **Fix 5** (§8): **structured** STALE-confirm dangling-referent validation, covering single facts *and* joins.
- **Fix 6** (§9): server-side, table-granular profiler allow/deny with a **required** restricted read-only role for production.

### 1.2 Non-goals (deferred to SP-1.6)

Group ownership + delegation; bulk confirmation ergonomics; the full multi-catalog **adapter registry** (SP-1.5 does the *keying* + a catalog-scoped single adapter, not a multi-adapter registry); governance-queue repair/re-route; fact-vocabulary expansion (data-class / sensitivity / retention facts); **per-risk-tier TTL** (no `risk_tier` source exists at the overlay layer until the Domain Catalog service, SP-1b — SP-1.5 does per-fact-*type* TTL only).

### 1.3 Why "before SP-3"

SP-3 consumes the overlay through the **read seam** — `resolve_fact` and the `fact_key` / catalog-reference shape — and does not care how a fact was confirmed. The items that would get frozen wrong into SP-3 are the read-path ones (freshness enforcement, the fact-serving contract) and the persistence reference model. Those are Fix 1 and Fix 2, sequenced first.

### 1.4 Execution model

Subagent-driven, TDD per task, adversarial review per task, keep the full suite green (currently 1080 passed). Eight tasks (§11).

## 3. Foundations

### 3.1 `OverlayConfig` (sealed server-side config)

Fixes 3, 4, and 6 all need configuration that a caller cannot forge. Define an `OverlayConfig` dataclass, constructed at deploy/`register_overlay` time (not from `cmd.args`), and read by the command handlers and the worker stages:

```
OverlayConfig:
  ttl_by_fact_type: Mapping[fact_type, timedelta]   # per-fact-type re-verify horizon
  ttl_default: timedelta                            # fallback (e.g. 180d)
  ttl_min, ttl_max: timedelta                       # clamp bounds (reject out-of-range config)
  ttl_jitter_fraction: float                        # e.g. 0.1 (±10%)
  renewal_grace: timedelta                          # pre-expiry window (e.g. 14d)
  drift_scan_interval: timedelta                    # how often drift runs (e.g. 15m)
  drift_freshness_sla: timedelta                    # max drift-watermark age before reads fail closed (>= interval, e.g. 60m)
  profiler_rules: tuple[ProfilerRule, ...]          # catalog_source + schema + table allow/deny (§9)
  profiler_require_restricted_role: bool            # True in production
```

**Ownership:** registered once via `register_overlay(config=...)` and resolved through a fail-closed accessor (mirroring `current_catalog_adapter`). **Missing/invalid config fails closed:** a command or stage that needs config it cannot resolve is denied/skips-loud, never defaulted silently. `ttl_default`/bounds are the only permitted fallbacks, and only for TTL resolution.

### 3.2 Canonical source-qualified catalog reference

The single source of truth for "which object a fact references" is the typed `CatalogObjectRef` (`catalog_source, object_kind, schema, table, column?`), already carried on `OVERLAY_FACT_PROPOSED.payload["catalog_object_ref"]`. The **source-less** `display_object_ref(ref)` string is presentation-only and MUST NOT be a persistence or dependency key.

**Rule:** every read-model / snapshot / dependency sink keys on the **pair** `(catalog_source, canonical_object_ref)` — stored as an explicit `catalog_source` column plus the canonical dotted `schema.table[.column]` (never the "from → to" join display string; joins index their structured endpoints). Enumerated sinks to convert (§5).

## 4. Fix 1 — Read-time freshness guard (findings #3 + BLOCKER round-1: time-expiry AND stale-drift)

**Problem.** Two read-path fail-opens, both letting `resolve_fact` serve a VERIFIED value it should not:

1. **Time-expiry (finding #3):** the VERIFIED branch ignores `expires_at`, so between `expires_at` passing and the async poller/projection catch-up, a value is served past expiry.
2. **Stale-drift (round-1 BLOCKER):** catalog drift is detected on a *slower cadence* (Fix 4, e.g. 15 min). Between drift scans, a column can be renamed/dropped and the fact's `expires_at` is still in the future and un-STALED — so `resolve_fact` serves a VERIFIED `availability_time` for a **nonexistent column**. This is not time-based; the time-expiry check cannot cover it. Leaving it open contradicts "correct before SP-3."

**Fix — resolve_fact fails closed on either condition.** In the VERIFIED branch, before serving:

- **Time-expiry:** if `expires_at is not None and expires_at <= now` → block with reason `expired_pending_reverify` (status REVERIFY), surfacing the current value as read-only `prior_value`.
- **Drift-freshness:** resolve the drift **watermark** for this fact's `catalog_source` (the persisted "last successful drift scan completed at", §7). If `now - watermark > OverlayConfig.drift_freshness_sla` (or no watermark exists) → block with reason `drift_stale_pending_scan`. This makes the overlay refuse to vouch for referents whose currency it cannot attest, and it *bounds* the drift window: SP-3 can only ground when the drift view is within SLA.

`now` becomes an optional trailing keyword on `resolve_fact` (defaults to `datetime.now(UTC)`) for deterministic tests and to share the poller's Python clock basis (compare in Python, not SQL — avoid a second clock source).

**Blocked-result context (round-1 MEDIUM).** Extend `_overlay_blocked`/`ResolvedFact` so a blocked result carries `expires_at`, `confirmed_event_id`, `catalog_source`, and the reason — SP-3 and operators need to distinguish "expired", "drift-stale", "reverify", "missing" and to debug.

**Seam impact (SP-3).** Backward-compatible optional `now` kwarg; a richer blocked result. SP-3 must treat any non-VERIFIED result (including REVERIFY-with-`prior_value` and the two new reasons) as **blocked, not usable** — the fail-closed contract already requires this. The drift-freshness guard also means SP-3 grounding implicitly requires a fresh drift checkpoint, which is the intended safety property.

**Interaction.** The drift-freshness half depends on Fix 4's watermark existing, so it sequences after Fix 4 (§11: task 5). The time-expiry half is standalone and sequences first (task 3).

**Tests.** VERIFIED past `expires_at` → blocked `expired_pending_reverify`. VERIFIED with a drift watermark older than the SLA (or none) → blocked `drift_stale_pending_scan`. VERIFIED, in-date, fresh drift → served. Blocked results carry `expires_at`/`confirmed_event_id`/`catalog_source`. All driven with injected `now`.

## 5. Fix 2 — Canonical source-qualified reference model + migration (findings #10 + round-1 BLOCKERs)

**Problem.** `fact_key` embeds `catalog_source`, but the source-less `display_object_ref` string leaks into every secondary sink, so a second catalog's identically-named object collides across the whole read surface, and drift/dependency cross-fires between catalogs.

**Fix — one canonical source-qualified model across ALL sinks.** Convert every place that stores or keys a catalog reference from the source-less string to `(catalog_source, canonical_object_ref)`. Enumerated sinks (round-1 BLOCKER #2):

- `overlay_fact_dependency.ref_object` → add `catalog_source`; key dependency lookups + `dependents_of` on the pair.
- `overlay_catalog_object` (drift snapshot) `object_ref` → add `catalog_source`; the anti-join and per-object upsert key on the pair (see §7 for the catalog-scoped anti-join).
- `overlay_fact_state.object_ref` and `overlay_proposal.object_ref` → add `catalog_source` (these read-model rows already have the structured ref available on their source events).
- `Change.object_ref` (drift change record) and the fingerprint/`_type_fingerprint` path → carry `catalog_source`.
- `CatalogObject.object_ref` / `PostgresCatalog.fingerprint` and the `FixtureCatalog` facts/owners maps → key on the pair.
- The `display_object_ref` string is retained **only** for human display; it is removed from every key/dependency/snapshot path.

**Migration / rebuild (round-1 BLOCKER #3 — "small and rebuildable" is not a plan).** The read-model rows cannot be backfilled from the source-less `object_ref`, but `catalog_source` is present in history (`OVERLAY_FACT_PROPOSED.payload["catalog_object_ref"]`) and in the adapter. So the migration is a **reset-and-rebuild**, exploiting that these are droppable, deterministically-rebuildable projections:

1. Ship the DDL adding `catalog_source` to `overlay_fact_dependency`, `overlay_catalog_object`, `overlay_fact_state`, `overlay_proposal` (and the change/snapshot tables), with the new composite keys.
2. **Reset** `overlay_fact_state`, `overlay_proposal`, `overlay_fact_dependency` and **reset the overlay projection checkpoint** to 0.
3. **Replay** the overlay event stream through the (updated) `OverlayProjection`, which now reads `catalog_object_ref` (structured, source-qualified) from each event and writes source-qualified rows.
4. **Rebuild** `overlay_catalog_object` (the drift snapshot) from the adapter (per `catalog_source`), not from the old rows.
5. A rebuild/replay acceptance test proves the read models are byte-identical-modulo-`catalog_source` and that two catalogs with identical `schema.table.column` no longer collide.

**Seam impact (SP-3).** This is the load-bearing seam change — SP-3 grounds against catalog-source-qualified references, not a single-catalog assumption. The `resolve_fact`/`fact_key` surface is already source-qualified; this aligns every secondary index and the blocked-result `catalog_source` (§4).

**Scope boundary.** SP-1.5 keeps the process-global **single** adapter but makes it **catalog-scoped-correct** (it carries its own `catalog_source`; all sink writes/reads are qualified by it). The multi-**adapter** registry (routing by `ref.catalog_source` across N live adapters) is SP-1.6.

**Tests.** Two facts on same-named objects in different `catalog_source`s do not collide in the dependency index, snapshot, or state; drift in one catalog does not stale the other's facts; replay from historical events yields source-qualified rows; projection rebuild reproduces the pair keys.

## 6. Fix 3 — Pre-expiry renewal + configurable TTL (finding #2 + round-1 MAJORs)

**Problem.** Re-verification can only begin *after* expiry (`_AWAITING_CONFIRMATION` excludes VERIFIED), so every fact takes a guaranteed outage; the TTL is a hard, un-jittered 180-day constant shared by single facts and joins.

### 6.1 Renewal timer (fully specified — round-1 MAJOR)

- **Kind:** a new timer kind `overlay_renewal`, admitted by a migration extending the `timers_kind_check` CHECK constraint (`0506`-style, additive + idempotent).
- **Arming:** when a fact is confirmed (single or join), arm `overlay_renewal` at `expires_at − OverlayConfig.renewal_grace` alongside the existing `overlay_expiry` timer. **Idempotency key:** `overlay_renewal:{fact_key}:{confirmed_event_id}` (bound to the exact confirmed version, so re-arming is idempotent and a superseded version's renewal is distinguishable).
- **Ownership:** a dedicated poller `fire_due_overlay_renewals` (mirroring `fire_due_overlay_expiries`), driven by a worker stage — NOT the generic timer→command bridge (like `overlay_expiry`, it needs to carry `fact_key`/`confirmed_event_id` and open a task). **Excluded** from `poll_due_timers`/the generic bridge exactly as `overlay_expiry` is.
- **Effect on fire:** CAS on the fact — if the fact is still VERIFIED at `confirmed_event_id` (not superseded, not already REVERIFY/STALE), open a **renewal re-verify task** (a distinct task marker/kind so §6.2 can recognize it) and mark the timer fired. If the fact was already superseded/expired/staled, the timer is a no-op (consumed) — the CAS makes a late renewal harmless.
- **After re-confirm:** re-confirming within grace produces a new `confirmed_event_id` and a fresh `expires_at`; the **old** `overlay_renewal` and `overlay_expiry` timers (keyed on the prior `confirmed_event_id`) are **cancelled** (mirroring `cancel_timers_for_task`), and new timers are armed for the new version. No stale renewal timer survives a renewal.

### 6.2 Confirmable-within-grace (precise CAS rule — round-1 MAJOR)

`confirm_fact` currently rejects VERIFIED (`_AWAITING_CONFIRMATION` excludes it). Extend it with a narrow, guarded renewal path — a VERIFIED fact is confirmable **iff all** hold:

1. an **open renewal task** exists for this `fact_key` (recognized by the renewal task marker/kind from §6.1 — distinguishing it from any other open task), AND
2. the confirm's `target_event_id == the fact's current `confirmed_event_id`` (CAS against the head — no confirming a stale view), AND
3. `now <= expires_at` (inside the live window; a past-expiry fact goes through the normal REVERIFY path, not renewal).

On success the fact stays VERIFIED (value replaced in place with a fresh `expires_at`) — **no VERIFIED→DRAFT regression** (the fold's `_NON_TERMINAL` rule is preserved). A VERIFIED fact with no open renewal task remains non-confirmable (unchanged). This makes renewal possible without opening a door to arbitrary VERIFIED re-confirmation.

### 6.3 TTL config (round-1 MAJOR)

- **Per fact type** from `OverlayConfig.ttl_by_fact_type`, falling back to `ttl_default`; clamped to `[ttl_min, ttl_max]` (a config value outside bounds is rejected at `register_overlay`, fail-closed).
- **Deterministic jitter:** `jitter = deterministic_hash(fact_key) mapped into ±ttl_jitter_fraction` — deterministic per fact (reproducible, testable, no `Math.random`), spreading an onboarding wave so facts do not expire in lockstep.
- **Per-risk-tier TTL is out of scope** (no `risk_tier` source at the overlay layer; deferred to SP-1b/Domain Catalog).
- **Missing config** for a fact type falls back to `ttl_default`; a totally-unresolvable config fails closed.

### 6.4 Dual-owner `approved_join` renewal (round-1 MAJOR)

The join path (`join_confirmation.py`) is separate from single-fact confirm and can append `PARTIALLY_CONFIRMED` before final validation (`:77`). Renewal must cover it:

- Renewal timers arm on a **fully** VERIFIED join (both owners confirmed), keyed on the join's `confirmed_event_id`, exactly as single facts.
- The confirmable-within-grace CAS rule (§6.2) applies to the join's re-confirmation; for a **dual-owner** join, renewal re-confirmation follows the same two-owner choreography as the original confirmation (both owners re-confirm; `PARTIALLY_CONFIRMED` is a legal intermediate state during renewal too), and only the final owner's confirm renews `expires_at`.
- **Same-owner** and **governance-side** joins follow their existing single-actor confirm choreography, with the renewal CAS/grace rules applied.

**Seam impact (SP-3).** None — the fact stays VERIFIED and servable throughout the grace window.

**Tests.** In-grace fact is served VERIFIED *and* has an open renewal task; re-confirm inside grace renews `expires_at` with no DRAFT regression and cancels the old timers; a renewal confirm racing the expiry timer resolves deterministically (CAS); a stale renewal timer after re-confirm is a no-op; TTL is per-fact-type + deterministically jittered and clamped; a dual-owner join renews only after both owners re-confirm.

## 7. Fix 4 — Crash-safe, multi-worker-safe drift driver (finding #1 residual + round-1 MAJORs)

**Problem.** `detect_catalog_changes` runs nowhere in production; and as written it (a) advances the snapshot *after* staling in **one** transaction — chunking it naively risks a crash advancing the snapshot before all dependents are staled, **permanently laundering drift**; (b) has no multi-worker guard; (c) does a global (un-catalog-scoped) snapshot anti-join; (d) has a dead `columns_fingerprint`.

**Fix.**

- **Worker stage on a slow cadence.** Add a drift stage to `run_worker_once`, gated by `OverlayConfig.drift_scan_interval` (not every ~1s tick), skipping cleanly when no adapter is wired, counted/logged via observability.
- **Multi-worker safety (round-1 MAJOR):** the drift stage acquires a **Postgres advisory lock** (or a leased `overlay_drift_run` row) before scanning, so only one worker scans at a time; cadence is driven by a **persisted last-run watermark** (`overlay_drift_watermark(catalog_source, last_completed_at, last_run_id)`), NOT an in-memory tick counter (which would duplicate scans after restart and race across workers). The watermark is exactly what Fix 1's drift-freshness guard reads (§4).
- **Crash-safe snapshot advance (round-1 MAJOR):** a drift run is a **durable, resumable unit**. Model it with an `overlay_drift_run(run_id, catalog_source, status, …)` record: (1) compute the change set; (2) stale dependents + open re-verify tasks in **chunks, each its own committed transaction**, recording progress; (3) advance the snapshot and write the watermark **only after all chunks commit** (status → `completed`). A crash mid-run leaves the snapshot un-advanced and the run `in_progress`/resumable — drift is never laundered (it is re-detected next run). The snapshot advance and watermark write are the last, atomic step.
- **Catalog-scoped anti-join (round-1 MAJOR):** `_save_snapshot`'s `DELETE FROM overlay_catalog_object WHERE object_ref <> ALL(%s)` becomes `… WHERE catalog_source = %s AND (catalog_source, object_ref) <> ALL(%s)` — scanning one catalog must never delete another catalog's snapshot rows.
- **Scale + correctness fixes:** batch the snapshot upserts (COPY / values-list, not per-object round-trips); implement a real `columns_fingerprint` so column-level drift (rename/type change) is actually detected; group re-verify tasks per (owner, change-set).

**Seam impact (SP-3).** None on reads directly, but the watermark it maintains is what Fix 1's drift-freshness guard gates grounding on.

**Tests.** A simulated dropped/renamed column stales dependents + opens tasks; column-level drift is detected (fingerprint no longer dead); a crash after some chunks but before snapshot-advance re-detects the drift next run (no laundering); two workers racing the drift stage do not double-scan (advisory lock); the anti-join scoped by `catalog_source` does not delete a second catalog's snapshot; the watermark advances only on completion.

## 8. Fix 5 — Structured STALE-confirm dangling-referent validation (finding #9 + round-1 MAJORs)

**Problem.** Confirming a REVERIFY/STALE fact re-blesses `state.prior_value` against only the JSON schema, never the live catalog (`confirmation_commands.py:96-106`); the snapshot then advances so the dangling name never re-triggers detection — a one-click VERIFIED fact pointing at a nonexistent column.

**Fix — structured validation against the live adapter (round-1 MAJOR: not string-based).** On confirm of a STALE/REVERIFY fact, extract the **structured** `CatalogObjectRef`s from the fact's value (not the display string) and validate each against the live adapter:

- **Object existence** (schema.table present in the adapter, for this `catalog_source`),
- **Column existence** (the referenced columns exist), and
- **Basic type compatibility** appropriate to the fact type — `availability_time.column` is a timestamp-like type; `scd.valid_from/valid_to` are date/timestamp; `approved_join.column_pairs` endpoints are join-compatible.

If any check fails, the default `prior_value` re-confirm is **rejected**; the owner must supply an **explicit value override** naming current object(s)/column(s). A clean re-confirm (all referents present + compatible) proceeds.

**Coverage (round-1 MAJOR):** applies on **both** the single-fact confirm path and the `approved_join` path — for the join, validate both endpoints and all `column_pairs` on both sides; cover partial confirmation (validate at each owner's confirm), same-owner joins, and governance-side joins. Lands **after Fix 2** so referents are source-qualified.

**Seam impact (SP-3).** None on reads; guarantees a re-confirmed fact's referents exist and are type-compatible at confirm time.

**Tests.** Confirming a STALE fact whose column was renamed is rejected unless overridden; a type-incompatible referent (availability column no longer timestamp) is rejected; a clean STALE re-confirm succeeds; a STALE `approved_join` with a renamed endpoint column is rejected on the owning side; the override path yields a VERIFIED fact pointing at live, compatible objects.

## 9. Fix 6 — Table-granular server-side profiler allowlist + required read-only role (finding #7 + round-1 MAJORs)

**Problem.** `allowed_schemas` is caller-supplied (`profiler_command.py:55`) — self-attesting; and schema-only granularity is weak for a bank; and read-only is only simulated via `SET LOCAL` on the full-privilege connection.

**Fix.**

- **Server-side, table-granular rules (round-1 MAJOR):** the allow/deny list moves to `OverlayConfig.profiler_rules` — `(catalog_source, schema, table)` allow/deny rules — sealed at `register_overlay`/deploy time, never from `cmd.args`. A `run_profiler` target is permitted only if it matches an allow rule and no deny rule; **missing/unresolvable config fails closed** (deny). Denials are audited (as today).
- **Required restricted read-only role for production (round-1 MAJOR):** for production acceptance, the profiler scan runs on a genuinely **restricted read-only DB role/connection** (least-privilege, cannot see out-of-scope schemas at the DB level). `OverlayConfig.profiler_require_restricted_role = True` in production; the `SET LOCAL transaction_read_only` simulation on the full-privilege connection is permitted **only** in tests/dev reference mode. The role contract (what it may read) is defined here; provisioning the role is a deployment step, but the code requires and checks it in production mode.

**Seam impact (SP-3).** None.

**Tests.** A `run_profiler` naming a `(catalog_source, schema, table)` outside the sealed rules is denied + audited even though the caller passed it; a denied table within an allowed schema is blocked (table granularity); missing config fails closed; production mode requires the restricted role (rejects/raises without it); dev/test mode accepts the `SET LOCAL` simulation.

## 10. Config & composition-root wiring (round-1 MEDIUM)

`register_overlay(config: OverlayConfig)` seals the config; a fail-closed accessor (`current_overlay_config()`, mirroring `current_catalog_adapter`) exposes it. Consumers: `confirm_fact`/renewal (TTL, grace), the renewal + drift worker stages (interval, SLA, watermark), the profiler command (rules, role requirement). The worker composition root (`worker.py:compose`) resolves the config the same way it resolves the catalog adapter (deployment-injected). No command or stage silently defaults; unresolved required config fails closed.

## 11. Task ordering and dependencies

Eight tasks; foundations first, then read-seam, then the rest, respecting dependencies:

1. **OverlayConfig surface + composition wiring** (§3.1, §10) — foundational for Fixes 3/4/6.
2. **Canonical source-qualified reference model + read-model migration/replay** (§3.2, §5) — foundational read-seam; precedes Fix 5 and the drift anti-join.
3. **Read-time time-expiry guard + richer blocked result** (§4 part 1) — read-seam; standalone; top priority.
4. **Crash-safe, multi-worker drift driver + watermark** (§7) — provides the watermark Fix 1's drift-guard needs.
5. **Read-time drift-freshness guard** (§4 part 2, round-1 BLOCKER) — depends on task 4's watermark.
6. **Pre-expiry renewal + configurable TTL + join coverage** (§6) — depends on task 1 (config).
7. **Structured STALE-confirm dangling-referent validation + join coverage** (§8) — depends on task 2 (source-qualified structured refs).
8. **Table-granular profiler allowlist + required read-only role** (§9) — depends on task 1 (config).

Minimum viable SP-3-unblocking slice: tasks 1–5 (the read seam + freshness). 6–8 harden the confirm/ops paths.

## 12. Testing strategy

All hermetic against the ephemeral-PostgreSQL harness (transactional `db`/`conn` fixtures). TDD each task RED→GREEN. Beyond the per-fix tests (§§4–9), **acceptance tests** (round-1):

- Source-qualified **replay** rebuilds the read models from old events (`catalog_object_ref` payload).
- **Two catalogs** with identical `schema.table.column` do not collide (dependency, snapshot, state, drift).
- Drift-run **crash before snapshot-advance** re-detects drift next run (no laundering).
- **Two workers** racing the drift stage do not double-scan.
- **Renewal-confirm vs expiry-timer race** resolves deterministically; **stale renewal timers** after re-confirm are no-ops.
