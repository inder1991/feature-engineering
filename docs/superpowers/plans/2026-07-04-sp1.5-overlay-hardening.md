## Task 2 — Canonical source-qualified reference model + reset-and-replay migration (§3.2, §5) — THE MIGRATION

**Why now:** load-bearing read seam; SP-3 grounds against source-qualified refs. Precedes Task 7 (structured validation) and Task 4's catalog-scoped anti-join.

**Grounded current state:** `overlay_fact_state.object_ref` / `overlay_proposal.object_ref` (source-less), `overlay_fact_dependency` PK `(fact_key, ref_object)` + `ref_object` index, `overlay_catalog_object` PK `object_ref`. `OverlayProjection.apply` writes `payload["object_ref"]` (source-less) but the event ALSO carries `payload["catalog_object_ref"]` (structured `asdict(CatalogObjectRef)` incl. `catalog_source`, decoded by `identity._ref_from_payload`). `_dependencies` builds source-less dotted strings.

**Files:** new migration `db/migrations/0920_overlay_source_qualified.sql`; modify `overlay/projection.py`, `overlay/catalog.py`, `overlay/catalog_changes.py` (`_save_snapshot` write only). Tests: `tests/featuregen/overlay/test_source_qualified.py` (new) + updates to projection/drift tests.

**Interfaces produced:**
- Migration adds `catalog_source text` to `overlay_fact_state`, `overlay_proposal`, `overlay_fact_dependency`, `overlay_catalog_object`; new PKs `overlay_fact_dependency (fact_key, catalog_source, ref_object)` and `overlay_catalog_object (catalog_source, object_ref)`; ref index → `(catalog_source, ref_object)`.
- `catalog.CatalogAdapter` gains `catalog_source` (property or method) so sinks qualify snapshot rows; `CatalogObject` unchanged (its `object_ref` stays the dotted canonical; the source is the adapter's).
- **F5 — adapter source-mismatch fails closed:** `get_fact`/`owner_of`/`get_object` on a `ref` whose `ref.catalog_source != adapter.catalog_source` return `None` (never a same-named object from another source). Today `FixtureCatalog` keys facts/owners by the SOURCE-LESS `display_object_ref` (`catalog.py:57`), so two catalogs' identically-named objects would cross-leak. Key the maps on `(ref.catalog_source, display_object_ref)` AND guard on the adapter's own source.
- `OverlayProjection.apply` reads `catalog_source` from `_ref_from_payload(payload["catalog_object_ref"])` (single ref → `.catalog_source`; join → `from_ref.catalog_source`, and dependency rows carry BOTH endpoints' sources) and writes it to every sink row.
- `_dependencies` returns `set[tuple[catalog_source, ref_object]]` (or writes `(fact_key, catalog_source, ref_object)` rows directly).

**Migration sequence (the review-this part).** One SQL file, order matters:
1. `TRUNCATE overlay_fact_state, overlay_proposal, overlay_fact_dependency, overlay_catalog_object;` — empty first so `ADD COLUMN ... NOT NULL` (no default) is legal and no stale source-less rows survive. Safe: these are event-rebuildable / adapter-rebuildable projections.
2. `ALTER TABLE ... ADD COLUMN catalog_source text NOT NULL;` on all four (empty tables).
3. Drop + recreate the composite PKs/indexes: `overlay_fact_dependency` PK → `(fact_key, catalog_source, ref_object)`, its ref index → `(catalog_source, ref_object)`; `overlay_catalog_object` PK → `(catalog_source, object_ref)`.
4. `UPDATE projection_checkpoints SET checkpoint_seq = 0, head_seq = 0 WHERE projection_name = 'overlay';` — force full replay through the UPDATED projection.
5. (Snapshot `overlay_catalog_object` stays empty; the Task-4 drift stage / `_save_snapshot` rebuilds it per `catalog_source` from the adapter. Note this in a comment.)

The replay itself is runtime: the worker's projection stage (or an explicit `run_projection(conn, OverlayProjection())` in the acceptance test) repopulates source-qualified rows from `checkpoint_seq=0`.

**F7 — read-outage gating (operational).** Between the migration's TRUNCATE and the projection replaying to head, `resolve_fact` fails closed for previously-VERIFIED facts (safe, but a read OUTAGE). So this migration is a **maintenance-window operation**, not a hot deploy: the deploy runbook (1) applies the migration, (2) drives the replay to head — `rebuild_projection(conn, OverlayProjection())` or `run_projection` until `projection_lag('overlay') == 0`, (3) health-checks `checkpoint_seq == head_seq` for `overlay`, and only THEN resumes read traffic. Document this in the migration file header + add a rebuild helper the runbook calls.

## Task 4 — Crash-safe, multi-worker drift driver + watermark (§7)

**Files:** migration `0925_overlay_drift_run.sql`; `overlay/catalog_changes.py` (catalog-scoped anti-join, real `columns_fingerprint`, chunked crash-safe run); `runtime/worker.py` (drift stage, cadence-gated by `drift_scan_interval`, advisory-locked, skip-loud when no adapter/config).

**F6 — drift-run schema + resume rules (the review-this part):**
```
overlay_drift_run(
  run_id text PRIMARY KEY, catalog_source text NOT NULL,
  status text NOT NULL CHECK (status IN ('in_progress','completed','abandoned')),
  change_set_digest text NOT NULL,          -- identity of the computed change set (deterministic hash)
  chunk_cursor bigint NOT NULL DEFAULT 0,    -- last committed chunk index (resume point)
  started_at timestamptz NOT NULL, heartbeat_at timestamptz NOT NULL, completed_at timestamptz NULL
)
overlay_drift_watermark(catalog_source text PRIMARY KEY, last_completed_at timestamptz NOT NULL, last_run_id text NOT NULL)
```
- **Run lifecycle:** advisory lock (per `catalog_source`) → find/create the active run: if an `in_progress` run's `heartbeat_at` is older than a lease timeout (e.g. `2 × drift_scan_interval`), it is **abandoned** (its snapshot was never advanced, so the drift is simply re-detected — never laundered) and a fresh run starts; otherwise resume it from `chunk_cursor`.
- **Chunk idempotency:** each chunk stales its dependents + opens re-verify tasks with the EXISTING idempotent writes (`open_reverify_task` is idempotent per (fact, change); `_mark_degraded`-style ON CONFLICT), then bumps `chunk_cursor` + `heartbeat_at` in the SAME committed transaction. Re-running a chunk (after a crash) is a no-op on already-staled facts.
- **Atomic finish:** ONLY after the last chunk commits, advance the snapshot (`_save_snapshot`, catalog-scoped anti-join below) + write the watermark + `status='completed'` in one final transaction. A crash before this leaves the snapshot un-advanced ⇒ next run recomputes the SAME change set (digest matches) and finishes it.

**F6 — bootstrap (so Task 5's guard doesn't block everything on day one):** on first deploy there is NO watermark, so the Task-5 drift-freshness guard would block ALL reads. A **bootstrap step** (in `seed_overlay_authz` or a dedicated `bootstrap_overlay_drift`) runs one drift scan (or seeds an initial snapshot + a `last_completed_at = now` watermark per configured `catalog_source`) so a fresh deploy has a valid, in-SLA watermark before read traffic starts. Document + test.

**Steps:**
- [ ] Failing tests: dropped/renamed column stales dependents + opens tasks; column-level drift detected (fingerprint no longer `""`); crash after some chunks but before snapshot-advance re-detects next run (no laundering — same `change_set_digest`); an abandoned stale-heartbeat run is superseded; two workers do not double-scan (advisory lock); anti-join scoped by `catalog_source` does not delete another catalog's snapshot; watermark advances only on completion; bootstrap seeds an initial in-SLA watermark.
- [ ] Implement per the schema/lifecycle above; catalog-scoped anti-join (`… WHERE catalog_source = %s AND (catalog_source, object_ref) <> ALL(%s)`); real `columns_fingerprint`.

## Task 8 — Table-granular profiler allowlist + required read-only role (§9) — depends on Task 1

**Files:** `profiler_command.py`; `overlay/config.py` (add `readonly_scan_conn_factory`).

**F2 — a genuine restricted role needs a SEPARATE scan connection (the current path can't work).** Today `_run_profiler` scans and writes on the SAME connection/transaction: it runs `run_profiler_scan` under `SET LOCAL transaction_read_only=on` inside a savepoint, then `ROLLBACK TO SAVEPOINT` to re-enable writes and writes evidence/proposals (`profiler_command.py:26-43`). A truly restricted read-only DB role **cannot** do the write phase, so a boolean flag alone is not implementable. Split it:
- **Scan phase** on a genuinely restricted read-only connection obtained from `OverlayConfig.readonly_scan_conn_factory` (a `Callable[[], DbConn]` provider, deployment-injected — least-privilege role that cannot see out-of-scope schemas). Produces the candidate proposals + evidence IN MEMORY only.
- **Write phase** on the command's normal (full-privilege) `conn`: apply the profiler-rule allow/deny gate, then write evidence + `propose_fact` for each surviving candidate, inside `execute_command`'s transaction (unchanged atomicity for the writes).
- `profiler_require_restricted_role=True` (production) REQUIRES `readonly_scan_conn_factory` to be configured and used — fail closed (raise) if absent. Dev/test mode (`False`) may fall back to the current in-transaction `SET LOCAL` simulation on the full-privilege `conn`.

**Steps:**
- [ ] Failing tests: `run_profiler` naming a `(catalog_source, schema, table)` outside `OverlayConfig.profiler_rules` denied + audited even if the caller passed it; denied table within an allowed schema blocked (table granularity); missing config fails closed; **F2** production mode raises without `readonly_scan_conn_factory` and runs the scan on the restricted connection; dev/test accepts the `SET LOCAL` simulation.

## Plan review round 2 — resolutions (design locked; mechanics → code-first)

Round 2 surfaced 2 grounding misses, 4 design decisions, and 3 implementation-mechanic items. **Lesson (again): stop pre-speccing IMPLEMENTATION MECHANICS in prose — they only ground against real code + tests. The plan locks the DESIGN decisions below; the mechanics are resolved code-first with RED→GREEN tests during the build.**

**Grounding fixes (the plan was WRONG):**
- **F1:** `open_reverify_task` → `gates.tasks.open_task` does a plain `INSERT (task_id=mint_id(...))` — **NOT idempotent** (verified). So drift chunk-retry would duplicate tasks/timers. Reverify AND renewal task opens need a REAL idempotency key (e.g. a unique `(fact_key, target_event_id, task_kind)` partial index on open tasks, `ON CONFLICT DO NOTHING`). Exact key + constraint designed in code (Task 4/6) with a "re-run a chunk opens no duplicate task" test.
- **F6:** the adapter protocol has NO `get_object` — it has `list_objects`/`get_fact`/`owner_of`/`fingerprint`. Object/column existence (Task 7 validation, F5 source-mismatch) uses `fingerprint()` / `list_objects()` keyed on the pair; `get_fact`/`owner_of` fail closed on a foreign source. Drop all `get_object` references.

**Design decisions (LOCKED):**
- **F2:** a VERIFIED fact with an EMPTY dependency set is treated as projection corruption → **fail closed** (`drift_stale_pending_scan` / a `no_freshness_proof` reason), never served.
- **F3:** the profiler read-only isolation must cover SCAN-TIME METADATA too — the scan runs against a **restricted CatalogAdapter built on the read-only connection**, not the deployment's (possibly privileged) adapter. `OverlayConfig` provides a restricted-adapter/read-only-conn factory; production requires it; the write phase uses the normal conn + adapter.
- **F4 (DECIDED — disallow):** SP-1.5 has ONE catalog adapter, so it **cannot** validate/attest a join endpoint in a different `catalog_source`. **Cross-catalog `approved_join`s are DISALLOWED in SP-1.5** — a join whose `from_ref.catalog_source != to_ref.catalog_source` is rejected at propose/confirm; same-source joins validate both endpoints against the single adapter.
