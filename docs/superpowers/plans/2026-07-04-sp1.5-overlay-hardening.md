# SP-1.5 — Metadata Overlay Hardening Implementation Plan

> **For agentic workers:** build code-first, TDD per task (RED→GREEN), keep the full suite green. Steps use checkbox (`- [ ]`). Ground every assertion in the actual code before writing — the SP-0.5 lesson: prose that isn't grounded doesn't converge.

**Goal:** Harden the SP-1 metadata-overlay READ path so it is correct, source-qualified, and freshness-guarded before SP-3 (grounding) freezes its seams.

**Spec:** `docs/architecture/2026-07-03-sp1.5-overlay-hardening-design.md` (the authority; this plan sequences it).

**Architecture:** Eight tasks; 1–5 are the SP-3-unblocking read seam (config → source-qualified model → time-expiry guard → drift driver → drift-freshness guard), 6–8 harden confirm/ops. Foundations first; the one migration (Task 2) is reset-and-replay against the event stream.

**Tech stack:** Python 3.11+, psycopg 3, PostgreSQL 15+ (ephemeral `pytest-postgresql`), `python-ulid`. Tests: `pytest`, transactional `db`/`conn` fixtures, 60s SIGALRM.

## Global constraints

- **Env:** `.venv/bin/python -m pytest ...` (no bare `python`/`timeout`). Ruff line-length 100; lint `E,F,I,UP,B,PIE`.
- **Migrations:** new DDL as a new lexically-ordered `db/migrations/NNNN_*.sql`; idempotent; the `schema_migrations` ledger checksums each file — NEVER edit a shipped migration. The overlay tables live in `0507_overlay_tables.sql` and timers in `0506_overlay_timers.sql`, but the CURRENT MAX migration is `0915_external_commands_backoff.sql`, so new SP-1.5 migrations take the next free numbers: **`0920`** (source-qualified, Task 2), **`0925`** (drift-run/watermark, Task 4), **`0930`** (renewal kind, Task 6). `ALTER TABLE overlay_*` in a later-numbered file is fine — migrations apply in lexical order and 0920 > 0507.
- **Fail-closed config:** all deploy config comes from `OverlayConfig` (Task 1, done) via `current_overlay_config()` — NEVER from `cmd.args`. A stage/command that cannot resolve required config denies / skips-loud.
- **Event-sourcing invariant:** overlay read models (`overlay_fact_state`, `overlay_proposal`, `overlay_fact_dependency`) are droppable + deterministically rebuildable from the overlay event stream via `OverlayProjection`. `overlay_catalog_object` (drift snapshot) is rebuilt from the ADAPTER, not events.
- **Reference model:** the canonical key is the PAIR `(catalog_source, canonical_object_ref)` where `canonical_object_ref = display_object_ref(single_ref)` = dotted `schema.table[.column]`. `display_object_ref` stays presentation-only for joins (`from -> to`); joins index their STRUCTURED endpoints, never that string.
- **Commit cadence:** one commit per task; prefix `feat(sp1.5):`/`fix(sp1.5):`; end body with `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>`.

## Task 2 — Canonical source-qualified reference model + reset-and-replay migration (§3.2, §5) — THE MIGRATION

**Why now:** load-bearing read seam; SP-3 grounds against source-qualified refs. Precedes Task 7 (structured validation) and Task 4's catalog-scoped anti-join.

**Grounded current state:** `overlay_fact_state.object_ref` / `overlay_proposal.object_ref` (source-less), `overlay_fact_dependency` PK `(fact_key, ref_object)` + `ref_object` index, `overlay_catalog_object` PK `object_ref`. `OverlayProjection.apply` writes `payload["object_ref"]` (source-less) but the event ALSO carries `payload["catalog_object_ref"]` (structured `asdict(CatalogObjectRef)` incl. `catalog_source`, decoded by `identity._ref_from_payload`). `_dependencies` builds source-less dotted strings.

**Files:** new migration `db/migrations/0920_overlay_source_qualified.sql`; modify `overlay/projection.py`, `overlay/catalog.py`, `overlay/catalog_changes.py` (`_save_snapshot` write only). Tests: `tests/featuregen/overlay/test_source_qualified.py` (new) + updates to projection/drift tests.

**Interfaces produced:**
- Migration adds `catalog_source text` to `overlay_fact_state`, `overlay_proposal`, `overlay_fact_dependency`, `overlay_catalog_object`; new PKs `overlay_fact_dependency (fact_key, catalog_source, ref_object)` and `overlay_catalog_object (catalog_source, object_ref)`; ref index → `(catalog_source, ref_object)`.
- `catalog.CatalogAdapter` gains `catalog_source` (property or method) so sinks qualify snapshot rows; `CatalogObject` unchanged (its `object_ref` stays the dotted canonical; the source is the adapter's).
- `OverlayProjection.apply` reads `catalog_source` from `_ref_from_payload(payload["catalog_object_ref"])` (single ref → `.catalog_source`; join → `from_ref.catalog_source`, and dependency rows carry BOTH endpoints' sources) and writes it to every sink row.
- `_dependencies` returns `set[tuple[catalog_source, ref_object]]` (or writes `(fact_key, catalog_source, ref_object)` rows directly).

**Migration sequence (the review-this part).** One SQL file, order matters:
1. `TRUNCATE overlay_fact_state, overlay_proposal, overlay_fact_dependency, overlay_catalog_object;` — empty first so `ADD COLUMN ... NOT NULL` (no default) is legal and no stale source-less rows survive. Safe: these are event-rebuildable / adapter-rebuildable projections.
2. `ALTER TABLE ... ADD COLUMN catalog_source text NOT NULL;` on all four (empty tables).
3. Drop + recreate the composite PKs/indexes: `overlay_fact_dependency` PK → `(fact_key, catalog_source, ref_object)`, its ref index → `(catalog_source, ref_object)`; `overlay_catalog_object` PK → `(catalog_source, object_ref)`.
4. `UPDATE projection_checkpoints SET checkpoint_seq = 0, head_seq = 0 WHERE projection_name = 'overlay';` — force full replay through the UPDATED projection.
5. (Snapshot `overlay_catalog_object` stays empty; the Task-4 drift stage / `_save_snapshot` rebuilds it per `catalog_source` from the adapter. Note this in a comment.)

The replay itself is runtime: the worker's projection stage (or an explicit `run_projection(conn, OverlayProjection())` in the acceptance test) repopulates source-qualified rows from `checkpoint_seq=0`.

**Steps:**
- [ ] **Step 1: failing acceptance test.** Two facts on identically-named objects in different `catalog_source`s (e.g. `pg:core` vs `pg:mart`, both `public.customers.signup_at`): after replay, `overlay_fact_state`/`overlay_fact_dependency` have TWO distinct rows keyed by source (no collision); `dependents_of` for one source does not return the other's fact. RED (currently they collide on the source-less string).
- [ ] **Step 2: migration SQL** per the sequence above. Run `.venv/bin/python -m pytest tests/projections -q` to confirm the schema-migration ledger accepts the new file.
- [ ] **Step 3: projection + `_dependencies` write `catalog_source`** from the structured payload ref.
- [ ] **Step 4: adapter exposes `catalog_source`**; `_save_snapshot` writes it (anti-join scoping deferred to Task 4).
- [ ] **Step 5: replay/rebuild test** — reset checkpoint, `run_projection` to head, assert source-qualified rows are byte-identical-modulo-`catalog_source` to a hand-built expectation; assert projection remains droppable+rebuildable.
- [ ] **Step 6:** full suite green; commit.

**Tests (spec §5):** two-catalog no-collision (dependency, snapshot, state); drift in one catalog does not stale the other's facts (partial — full anti-join is Task 4); replay from historical `catalog_object_ref` yields source-qualified rows; rebuild reproduces the pair keys.
