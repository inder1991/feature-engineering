### Task 2: Prove-health-on-resolve + generic audit + rebuild clears degraded

**Findings B1-resolve + run-specific + rebuild + review-round-2 (B1-audit-API, B3-registry, clear-too-much, advance-past-loop).**

**Design decisions (grounded in verified APIs):**
1. **Prove health before clearing (review B1):** `resolve_degraded_command` re-runs the affected projection *past the poison* first via a precise helper (below). Unblock ONLY when it advances past `poison_seq`; else refuse (marker stays).
2. **`advance_projection_past(conn, projection, poison_seq) -> bool` (review r2 — "run once" is insufficient; `run_projection` is batched):** loop `run_projection(conn, projection)` until either `read_as_of(conn, projection.name) >= poison_seq` → return `True` (healthy — **self-verify: `>=`, not `>`; the poison at seq P is consumed once the checkpoint REACHES P**, and in the common single-poison-at-tail setup remediation advances the checkpoint to exactly P), OR a call makes **zero progress** (checkpoint unchanged — still stuck at this poison or an EARLIER one in the shared ordered stream) → return `False`. Bound the loop (≤ 1000 iters). Being blocked by an earlier aggregate's poison correctly yields `False`.
3. **Aggregate-generic audit via the REAL `record_security_event` API (review r2 — NO `action` column; signature is `event_type`/`attempted_action`/`decision`; self-verify: `decision` has a CHECK constraint `('denied','allowed_break_glass','flagged')` — `"allowed"` would raise CheckViolation and crash the resolve):** `record_security_event(conn, event_type="DEGRADED_RESOLVED", actor=cmd.actor, attempted_action="resolve_degraded", decision="flagged", aggregate=<agg>, aggregate_id=<id>, reason=<reason>)`. Works for `run`/`feature`/`request`/`overlay_fact`; not a domain event.
4. **Clear the SPECIFIC marker (review r2 — key includes `projection_name`):** delete by `(projection_name, aggregate, aggregate_id)`, NOT `(aggregate, aggregate_id)` — if two projections degrade the same aggregate, proving one healthy must not clear the other. So `resolve_degraded_command` takes `projection_name` from `cmd.args` (or resolves each degraded projection separately).
5. **Projection lookup seam (review r2 B3 — must NOT call `worker.compose()`, which has registration side effects):** add `src/featuregen/projections/registry.py` — `register_projection_for_repair(name, projection)` / `projection_for_repair(name) -> Projection | None`. The composition root registers its projections there; `resolve_degraded_command` looks up by name with no side effects.
6. **Rebuild clears `projection_degraded`** (by `projection_name`) before replay.
7. **Transaction:** `resolve_degraded_command` runs within `conn.transaction()` (it calls `run_projection`, which uses SAVEPOINT — must be in a transaction; see the autocommit rule).

**Files:** Create `src/featuregen/projections/registry.py`. Modify `src/featuregen/aggregates/run_lifecycle.py` (`resolve_degraded_command`), `src/featuregen/projections/runner.py` (`advance_projection_past`, `rebuild_projection`), `src/featuregen/runtime/worker.py` (`compose()` registers projections for repair). ## Workstream 2 — Worker wiring

### Task 3: External-command dispatch stage (claimed, concurrency-safe, autocommit)

**Findings + B2 concurrency + MED shared-connection.** `dispatch_command` exists but no worker stage calls it. A naive unlocked `SELECT pending … then dispatch` lets two workers hand the SAME row to `dispatch_command`, whose recovery path can **re-invoke** (it treats an already-`dispatched` row as crash recovery). And `dispatch_command` commits internally, so it must run on the worker's **autocommit** connection.

**Files:** Modify `src/featuregen/runtime/worker.py` (stage), `src/featuregen/runtime/external_commands.py` (a `claim_pending_external(conn, owner, batch)` helper + a caller registry). Test: `tests/featuregen/runtime/test_worker.py` (autocommit-connection test).

**Design (review r2 B2 — `external_commands` has NO lease columns; "lease by status='dispatched'" makes `dispatch_command` treat it as crash-recovery and possibly re-invoke).** Do the atomic pick+claim in ONE statement, and invoke it as a *fresh* dispatch (not recovery):

- `claim_next_pending(conn) -> str | None` — one atomic statement: `UPDATE external_commands SET status='dispatched', dispatched_at=%s, attempts=attempts+1 WHERE command_id = (SELECT command_id FROM external_commands WHERE status='pending' ORDER BY created_at FOR UPDATE SKIP LOCKED LIMIT 1) RETURNING command_id, integration, request_payload`. `SKIP LOCKED` guarantees no two workers claim the same pending row; it returns the row it claimed. This IS "move the SKIP LOCKED claim into the claim step" (review r2's preferred option), using the existing status set — no schema change.
- To avoid `dispatch_command`'s claim-vs-recovery ambiguity (it re-reads `status` and would see `dispatched` → recovery), the worker calls a **fresh-invoke** entrypoint, `invoke_claimed_external(conn, claimed_row, caller, *, now)`, that runs `dispatch_command`'s Step-2/Step-3 (external call + finalize) for a row IT just claimed as first-dispatch — reusing `dispatch_command`'s call/finalize logic but skipping the re-claim. Genuine crash-recovery (a `dispatched` row from a dead worker, found later) stays the province of `dispatch_command`'s existing recovery path, unchanged. **(If cleanly separating fresh-invoke from `dispatch_command` proves infeasible without duplicating logic, the fallback is to add a `lease_owner text` column + a stale-lease reclaim, and distinguish fresh vs recovery by owner — flag this to the reviewer; prefer the no-schema-change split.)**

**Files:** Modify `src/featuregen/runtime/external_commands.py` (`claim_next_pending`, `invoke_claimed_external`, caller registry), `src/featuregen/runtime/worker.py` (stage on the autocommit connection). Test: `tests/featuregen/runtime/test_worker.py` (throwaway-autocommit-DB), `tests/featuregen/runtime/test_external_commands.py`.

**Interfaces:**
- Produces: `claim_next_pending(conn) -> ClaimedExternal | None` (atomic, SKIP LOCKED); `invoke_claimed_external(conn, claimed, caller, *, now)`; `register_integration_caller(name, caller)`/`current_integration_callers()` (fail-closed accessor). ### Task 6: Timer-actor attestation + `validate_identity` (keep attestation mandatory) + token-edge guard

**Findings + M weakening + M test-API.** `_TIMER_RUNTIME_ACTOR` is `auth_method="internal"`, **no attestation**, but `validate_identity` requires a service actor to be attested — so it would be rejected. Admitting *all* authenticated `internal` envelopes (my first idea) would relax the validator: `IdentityEnvelope` is plain data, so validate can't know an envelope came from `mint_trusted_identity`. And my sample test used a non-existent `build_service_identity(auth_method=…)` API.

**Design decision (keep attestation mandatory — resolves the weakening):** the internal trust root must carry a **positive attestation marker** (e.g. `attestation="internal-trust-root:timer-runtime"`), so `validate_identity` keeps requiring attestation for **every** service actor (an unattested `internal` envelope is still REJECTED — no weakening). Only the `auth_method` check is broadened to admit `internal` alongside `workload-identity`. So the rule is uniform: *service actors are always attested; `internal` is a distinct, attested trust-root method.*

### Task 7: Transaction boundary for `execute_command` (DECIDED) + advisory-lock sites open a transaction

**Findings B3 + M call-sites.** The plan must DECIDE `execute_command`'s transaction boundary (it determines idempotency, exception cleanup, whether `_pending` rows strand). And making `append_event`/`record_security_event` *raise* outside a transaction would break the many direct callers (tests + domain functions) that don't go through `execute_command`.

**Decided transaction design for `execute_command` (review r2 — the CURRENT `_claim` does NOT commit; it relies on `ON CONFLICT DO NOTHING` where the loser BLOCKS on the winner's uncommitted row within the caller's transaction. A "committed claim" would break that mechanism AND commit a caller's/test-fixture's broader transaction).** So make `execute_command` **transaction-agnostic**, not self-committing:
1. Wrap the WHOLE body (claim → replay/takeover → authz → degraded-check → dispatch → `_finalize`/`_release`) in a single `with conn.transaction():`. Under an autocommit connection this is one real transaction (atomic — no partial idempotency/side-effect commits); under a caller that already holds a transaction it is a **savepoint** (nested — preserves the existing "runs inside the caller's transaction" behavior and the `ON CONFLICT`-blocks-loser concurrency).
2. `execute_command` NEVER calls `conn.commit()` itself — the caller (or the autocommit boundary) owns the final commit. This keeps every existing transactional-fixture caller working unchanged.
3. On a handler exception the transaction/savepoint rolls back, which **automatically un-inserts the `_pending` claim** (no explicit committed release needed) — a retry re-claims. The existing `_release` for denials/degraded stays (it deletes the claim within the transaction).

**Advisory-lock sites (non-breaking):** `append_event` and `record_security_event` **ensure a transaction** rather than raise — wrap their advisory-lock+insert in `with conn.transaction():` (a savepoint if already in a tx, a real transaction if the connection is autocommit). This keeps `pg_advisory_xact_lock` held through the dependent insert on ANY connection, and does not break the many direct callers.

**Files:** Modify `src/featuregen/commands/api.py` (`execute_command`), `src/featuregen/events/store.py` (~101), `src/featuregen/security/audit.py` (~126). Tests: `tests/featuregen/commands/`, `tests/featuregen/events/`, `tests/featuregen/security/`.

- [ ] **Step 1: Failing tests** — (a) `execute_command` on an **autocommit** connection where the handler raises after claim: NO partial side effect / `_pending` remains committed (the whole body rolled back atomically), and a retry re-claims. (b) `append_event` / `record_security_event` called on an **autocommit** connection (no open tx) still produce a correct chained/gapless row (the advisory lock held through the insert) — RED today if the lock releases early. (c) an existing in-transaction caller (transactional `db` fixture) still works unchanged (the wrap is a savepoint) — the whole suite is the guard.
