### Task 2: Prove-health-on-resolve + generic audit + rebuild clears degraded

**Findings B1-resolve + run-specific + rebuild.** Deleting `projection_degraded` and unblocking WITHOUT proving the poison is gone leaves a window where commands run until the next tick re-marks (a fail-open). A run-scoped `DEGRADED_RESOLVED` event cannot correctly audit a degraded `feature`/`request`/`overlay_fact` aggregate. `rebuild_projection` doesn't clear `projection_degraded`.

**Design decisions (resolve two blockers coherently):**
1. **Prove health before clearing (review B1):** `resolve_degraded_command` re-runs the affected projection *past the poison* first. If the projection still cannot advance past `poison_seq`, resolve **fails** (`accepted=False`, "still degraded: remediate the root cause") and the marker STAYS. Only when the projection advances past the poison does resolve delete the `projection_degraded` row. No unblock without proven health.
2. **Aggregate-generic audit, not a domain event (review run-specific):** "degraded" is an *operational/processing* fact, not a domain fact of the feature/run — so the remediation is recorded in the aggregate-generic **`security_audit`** detective-control log (actor, timestamp, aggregate, aggregate_id, reason), NOT a per-aggregate domain event. This works for `run`/`feature`/`request`/`overlay_fact` alike and avoids event-sourcing a non-domain fact. (Supersedes the earlier "DEGRADED_RESOLVED event" idea.)
3. **Rebuild clears `projection_degraded`** before replay — the runner re-derives it if the poison persists.

**Files:** Modify `src/featuregen/aggregates/run_lifecycle.py` (`resolve_degraded_command`), `src/featuregen/projections/runner.py` (`rebuild_projection` + a `advance_past` helper), reuse `security/audit.py:record_security_event`. Tests: `tests/featuregen/aggregates/test_run_lifecycle.py`, `tests/projections/test_rebuild.py`, `tests/projections/test_fail_closed.py`.

**Interfaces:**
- Consumes: a projection registry (to look up the `Projection` by `projection_degraded.projection_name`); `run_projection`; `record_security_event`.
- Produces: `resolve_degraded_command` — re-run the marked projection; if it advances past `poison_seq` delete the marker + `record_security_event("degraded_resolved", …)` and `accepted=True`; else leave the marker + `accepted=False`. `rebuild_projection` deletes `projection_degraded` for the projection before replay.

### Task 7: Transaction boundary for `execute_command` (DECIDED) + advisory-lock sites open a transaction

**Findings B3 + M call-sites.** The plan must DECIDE `execute_command`'s transaction boundary (it determines idempotency, exception cleanup, whether `_pending` rows strand). And making `append_event`/`record_security_event` *raise* outside a transaction would break the many direct callers (tests + domain functions) that don't go through `execute_command`.

**Decided transaction design for `execute_command`:** mirror the dispatcher's committed-claim + wrapped-step pattern:
1. **Claim** (`_claim`) writes the `_pending` idempotency marker and **commits on its own** — so concurrent duplicates see it (unchanged concurrency semantics).
2. **Body** (authz → degraded-check → handler dispatch → `_finalize`) runs inside a single `with conn.transaction():` so it is atomic regardless of the caller's autocommit state.
3. **On body failure:** the transaction rolls back AND the `_pending` claim is **released** (deleted / marked released) in its own committed statement, so a retry can re-claim — no stranded `_pending`. **On success:** `_finalize` replaces `_pending` with the real result inside the body transaction.

**Advisory-lock sites (non-breaking):** `append_event` and `record_security_event` **ensure a transaction** rather than raise — wrap their advisory-lock+insert in `with conn.transaction():` (a savepoint if already in a tx, a real transaction if the connection is autocommit). This keeps `pg_advisory_xact_lock` held through the dependent insert on ANY connection, and does not break the many direct callers.

**Files:** Modify `src/featuregen/commands/api.py` (`execute_command`), `src/featuregen/events/store.py` (~101), `src/featuregen/security/audit.py` (~126). Tests: `tests/featuregen/commands/`, `tests/featuregen/events/`, `tests/featuregen/security/`.

- [ ] **Step 1: Failing tests** — (a) `execute_command` on an **autocommit** connection where the handler raises after claim: the `_pending` marker is released (a retry can re-claim), and no partial side effect committed. (b) `append_event` / `record_security_event` called on an **autocommit** connection (no open tx) still produce a correct chained/gapless row (the advisory lock is held through the insert) — RED today if the lock releases early.

### Task 8: Bounded, observable handler-timeout connection leak
