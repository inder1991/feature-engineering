## DEFERRED to SP-1.6 (real infrastructure / feature scope, decided out of SP-1.5)

### D1 — Restricted-role profiler scan connection (`readonly_scan_conn_factory`)
- **State:** The flag `profiler_require_restricted_role` is **already handled correctly per the spec**
  (plan line 184: "REQUIRES `readonly_scan_conn_factory` ... fail closed (raise) if absent"). When
  the flag is `True` and no factory exists, the profiler **fails closed** — which is exactly what the
  spec mandates. `SET LOCAL transaction_read_only` remains the dev/test (`False`) path.
- **SP-1.6 work:** add `readonly_scan_conn_factory` to `OverlayConfig`; `_run_profiler` derives a
  restricted least-privilege connection/adapter for the SCAN phase (metadata + data visibility, not
  just write-prevention); production sets the flag `True`. Lifecycle: the factory owns the conn, the
  caller closes it in `try/finally` (closed on error too).
- **Why deferred:** needs a real DB-role provisioning + a connection-factory lifecycle — new
  infrastructure beyond SP-1.5's hardening budget. Fail-closed-when-absent is safe in the meantime.

### D2 — Chunked / resumable drift-run driver (`overlay_drift_run`)
- **State:** SP-1.5 ships an **atomic single-transaction** drift scan (stale + snapshot + watermark
  in one tx). It is **crash-safe** (a crash re-detects, never launders) and correct for the current
  catalog size. `0925` documents the deferral.
- **SP-1.6 work:** `overlay_drift_run` (run id, chunk cursor, heartbeat, digest), abandon/resume on
  a recomputed-digest mismatch (restart from chunk 0 if the catalog moved), inspectable progress —
  the large-catalog / long-scan design in the plan.
- **Why deferred:** a scale optimization; the atomic run is safe until catalogs are large enough that
  one transaction is impractical.

### D3 — Dual-owner `approved_join` pre-expiry renewal
- **State:** SP-1.5 **disallows** in-place renewal of a VERIFIED dual-owner join (`confirm_fact`
  denies it) and **excludes** them from the renewal poller. This AVOIDS the
  VERIFIED→PARTIALLY_CONFIRMED regression + laundering window that a naive two-step renewal opens.
  Consequence: dual-owner joins still take the expiry→reverify path (a brief re-verification outage),
  which SP-1.5's renewal was meant to remove for single-owner facts.
- **SP-1.6 work:** a safe dual-owner pre-expiry renewal — e.g. a dedicated renewal aggregate state
  that collects both owners' re-confirmations WITHOUT regressing the live VERIFIED status, then flips
  atomically; or a shadow re-verification that supersedes the live fact only on completion.
- **Why deferred:** doing it safely needs new state-machine design; the disallow is a correct,
  no-security-hole scope reduction (single-owner facts — the common case — get the no-outage renewal).

### D4 — `StubCatalog` F5 fail-closed parity (test fidelity)
- **State:** production `FixtureCatalog`/`PostgresCatalog` fail closed on a foreign `catalog_source`;
  the shared test double `StubCatalog` does not, so ~45 command tests pair it with other-source refs.
  The F5 boundary IS covered by `FixtureCatalog` in `test_source_qualified.py`.
- **SP-1.6 work:** add the guard to `StubCatalog` and align each consuming test's `catalog_source`.
- **Why deferred:** mechanical test churn disproportionate to a test-fidelity MINOR.

### D5 — Read/API-process `OverlayConfig` sealing
- **State:** `run_forever` (worker) seals the config at boot, so drift/renewal/profiler are active.
  `resolve_fact` has **no production caller yet** (SP-2+ consumers aren't wired), so the read-path
  config requirement isn't reachable in production today.
- **SP-1.6 work:** when SP-2 wires a read/API process that calls `resolve_fact`, that process must
  seal the config via `overlay_config_from_env()` at boot (mirroring the worker), and a startup
  assertion should fail closed if it doesn't.
- **Why deferred:** the process it applies to does not exist yet.
