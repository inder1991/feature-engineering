# SP-0.5 Backbone Hardening Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the 2026-07-02 review's blocker/major findings on the *built* SP-0 backbone so its headline guarantees (attested identity, tamper-evident audit, durable runtime that actually runs, deterministic-rebuild projections, dual-control) are enforced by mechanism rather than convention — before SP-3 consumers freeze these seams.

**Architecture:** The SP-0 kernel is correct but has four classes of gap: (a) a handler exception bypasses the DLQ and retries forever (`dispatch.py`), (b) nothing *runs* the outbox relay / timer poller / projection runner / queue worker in production, (c) audit and event immutability rest on convention (no write-once trigger, keyless hash chain), and (d) identity is self-asserted. This plan fixes them in dependency order: poison-handling and retry hygiene first (so a running daemon is safe), then the daemon, then immutability/audit, then the identity boundary. Every change is TDD, grounded in the existing event-sourced patterns.

**Tech Stack:** Python 3.11+, psycopg 3 (`psycopg[binary]`), PostgreSQL 15+ (tests boot an ephemeral cluster via `pytest-postgresql`), `python-ulid`, `jsonschema`. New runtime dep: `PyJWT[crypto]` (Task 10). Tests: `pytest` with the transactional `db` fixture; per-test 60s SIGALRM timeout.

## Global Constraints

- **Python floor:** `requires-python = ">=3.11"`; target `py311`. Ruff line-length 100 (E501 ignored — formatter owns it); lint select `E,F,I,UP,B,PIE`.
- **DB access:** all SQL via `psycopg` cursors; JSON columns via `psycopg.types.json.Json`/`Jsonb`. Never build SQL by string-concatenating values — always parameterize.
- **Event-sourcing invariant (do not violate):** every *derived read-model* state change (e.g. the `run_workflow_state.degraded` flag, deprecation state) must derive from an appended event applied by a projection — Task 8 removes the two places that mutate those directly. **Exception — CAS slot tables:** `feature_active_versions` is a command-owned compare-and-swap slot table that the established, reviewed `activate` path writes directly at command time (`activation.py:104-111`); `supersede`/`deactivate` mirror that pattern (Tasks 3/4) and their direct CAS write is correct, not a violation. The rule bites derived read models, not the activation slot.
- **Migrations:** add DDL as a new `src/featuregen/db/migrations/NNNN_*.sql` file (lexically ordered, loaded by `_sql_file_migrations()`); use `CREATE ... IF NOT EXISTS` / `CREATE OR REPLACE` so `apply_migrations` stays idempotent. Task 9 adds the ledger; until then, keep that idempotency.
- **IDs:** mint with the existing `mint_id("<prefix>")` / `f"{prefix}_{ULID()}"` convention already used in the codebase; never invent a new id scheme.
- **Tests:** use the existing `db` fixture (transactional, rolled back per test) and helpers in `tests/featuregen/_helpers.py` / the nearest `conftest.py`. A test that needs true cross-connection durability (relay/daemon) opens its own `psycopg.connect(db.info.dsn)` like `test_dispatch.py` already does.
- **Commit cadence:** one commit per task minimum, message prefix `fix(sp0.5):` (behavioural) / `feat(sp0.5):` (new capability) / `chore(sp0.5):`. End every commit body with the `Co-Authored-By` trailer.

### Task 1: Poison-message guard — a raising handler DLQs instead of retrying forever

**Finding:** BLOCKER #2. `process_one`'s outer `with conn.transaction()` wraps claim→handle→commit; any exception other than `HandlerTimeout`/`ConcurrencyError` (e.g. `registry.get` KeyError at `dispatch.py:146`, or a handler that raises) unwinds the outer tx, discarding `claim_one`'s `attempts+1` — so the row returns to `ready` unchanged and retries forever, wedging its partition.

**Files:**
- Modify: `src/featuregen/runtime/dispatch.py:135-195`
- Test: `tests/featuregen/runtime/test_dispatch.py`

**Interfaces:**
- Consumes: `claim_one`, `fail_retryable`, `fail_permanent`, `complete` (from `runtime/queue.py`), `HandlerRegistry.get` (raises `KeyError`), `ProcessOutcome`.
- Produces: `process_one(...) -> ProcessOutcome` — unchanged signature; new guarantee: an unknown handler or a raising handler results in `status="retryable"` (bumped toward DLQ) or `status="permanent"`, **never** a propagated exception, and the `attempts` increment is durably committed.

- [ ] **Step 3: Add the poison guard**

In `dispatch.py`, resolve the handler *inside* a guard and wrap the whole handle/commit body so any unexpected exception durably fails the claim. Replace the `handler = registry.get(claim.handler)` line and the `try/finally` block (currently `dispatch.py:146-195`) with:

```python
        # Resolve the handler under the poison guard: an unknown handler name is a
        # DETERMINISTIC failure (never retryable) — route straight to DLQ (review BLOCKER #2).
        try:
            handler = registry.get(claim.handler)
        except KeyError as exc:
            fail_permanent(conn, claim.id, error=str(exc))
            return ProcessOutcome(status="permanent", message_id=claim.message_id, queue_id=claim.id)

        handler_conn = handler_conn_factory(conn)
        timed_out = False
        try:
            ctx = _build_context(conn, claim, document_loader, handler_conn=handler_conn)

            try:
                result = _run_with_timeout(handler, ctx)
            except HandlerTimeout as exc:
                timed_out = True
                fail_retryable(conn, claim.id, error=str(exc))
                return ProcessOutcome(
                    status="retryable", message_id=claim.message_id, queue_id=claim.id
                )

            if result.disposition == Disposition.OK:
                try:
                    with conn.transaction():
                        commit_step(
                            conn,
                            ctx,
                            result,
                            message_id=claim.message_id,
                            expected_version=ctx.triggering_event.stream_version,
                            table_version=ctx.triggering_event.table_version,
                        )
                except ConcurrencyError as exc:
                    fail_retryable(conn, claim.id, error=f"OCC: {exc}")
                    return ProcessOutcome(
                        status="retryable", message_id=claim.message_id, queue_id=claim.id
                    )
                complete(conn, claim.id)
                return ProcessOutcome(status="ok", message_id=claim.message_id, queue_id=claim.id)

            if result.disposition == Disposition.RETRYABLE:
                fail_retryable(conn, claim.id, error=result.error or "retryable")
                return ProcessOutcome(
                    status="retryable", message_id=claim.message_id, queue_id=claim.id
                )

            fail_permanent(conn, claim.id, error=result.error or "permanent")
            return ProcessOutcome(
                status="permanent", message_id=claim.message_id, queue_id=claim.id
            )
        except (KeyError, Exception) as exc:  # noqa: BLE001 — the poison backstop
            # A handler that raised (or _build_context that could not resolve the triggering
            # event) is treated as a TRANSIENT delivery failure: bump attempts + backoff, DLQ
            # at the budget. This is what prevents the infinite poison-retry (review BLOCKER #2).
            # ConcurrencyError is handled above; HandlerTimeout returned above; anything here is
            # an unexpected fault, retried a bounded number of times then DLQ'd.
            fail_retryable(conn, claim.id, error=f"handler fault: {exc!r}")
            return ProcessOutcome(
                status="retryable", message_id=claim.message_id, queue_id=claim.id
            )
        finally:
            if not timed_out:
                handler_conn.close()
```

Note: `fail_retryable`/`fail_permanent` run on `conn` inside the outer `with conn.transaction()`, so their `attempts`/`status` write commits with the outer tx — the increment is now durable on the failure path.

```python
@dataclass(frozen=True, slots=True)
class WorkerTick:
    queue_processed: int
    relay_published: int
    timers_fired: int
    overlay_expiries: int
    projections_advanced: int
    reclaimed: tuple[int, int]
    errors: int


def run_worker_once(conn, registry, projections, *, owner, now, batch=50) -> WorkerTick:
    reclaimed = recover_stuck(conn)  # return expired leases first
    q = 0
    for _ in range(batch):
        outcome = process_one(conn, registry, owner=owner)
        if outcome.status == "idle":
            break
        q += 1
        if outcome.status in ("retryable", "permanent"):
            counters.incr("queue.fail")
    published = relay_publish_batch(conn, make_queue_publisher(), owner=owner)
    fired = 0
    for tid in poll_due_timers(conn, owner=owner, lease_seconds=30, batch=batch, now=now):
        if fire_timer(conn, tid, now=now).fired:
            fired += 1
    overlay = fire_due_overlay_expiries(conn, now=now)
    advanced = sum(run_projection(conn, p) for p in projections)
    return WorkerTick(q, published, fired, overlay, advanced, reclaimed, errors=0)
```
