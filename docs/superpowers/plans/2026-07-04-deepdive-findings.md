## 1. [BLOCKER/concurrency] src/featuregen/intake/commands.py:742  (concurrency-tx)
**The platform-wide global_seq advisory lock is held across an external LLM network call: submit_intent appends events (REQUEST_CREATED/RUN_CREATED/INTENT_SUBMITTED) and then calls call_llm inside the SAME execute_command transaction, so pg_advisory_xact_lock(_GLOBAL_SEQ_LOCK_KEY) (events/store.py:112, held until top-level commit) blocks EVERY event append in the system for the full LLM latency.**

- Scenario: Interleaving: (1) User A runs submit_intent via execute_command; on the autocommit API connection the whole body is one transaction (commands/api.py:100-102). (2) create_request_command appends REQUEST_CREATED -> append_event acquires pg_advisory_xact_lock(4_201_873_355_201_001); being xact-scoped it is held until the command's top-level COMMIT. (3) The same transaction reaches _produce_draft -> call_llm -> drive_structured_call (intake/llm.py:518), an HTTPS call to the LLM provider taking 10-120s (or hanging until client timeout, or indefinitely with no timeout). (4) Meanwhile every other append_event on every connection — every worker commit_step, every other command, every overlay/expiry append — blocks inside 'SELECT pg_advisory_xact_lock(...)'. The entire event store is frozen for the duration of one user's LLM call; concurrent submit_intents serialize end-to-end; the worker tick stalls mid-stage. A hung LLM connection freezes all writes platform-wide. The same pattern exists in fire_due_overlay_expiries: _apply_expiry appends OVERLAY_FACT_EXPIRED then calls resolve_authority against the (potentially remote) catalog adapter while holding the lock (overlay/expiry.py:68-88). Also creates an inconsistent lock-order pair with the security-chain lock (intake/llm.py:534 takes the chain lock after the global lock; other paths take chain first), a Postgres-detected deadlock window under two concurrent LLM auth failures.
- Fix: Never perform network I/O in a transaction that has already appended an event. Split submit_intent: commit the intent-opening events first (or record intent and hand the LLM normalization to an async worker step), then run call_llm outside the appending transaction and append DRAFT_CONTRACT_PRODUCED in a fresh transaction with the X4 folded-head CAS it already threads. Alternatively replace the global allocation lock with a gap-tolerant projection cursor (e.g. txid_snapshot/committed-visibility watermark) so append_event no longer serializes platform-wide. Audit all sites doing network calls after an append in the same tx (intake call_llm sites, overlay _apply_expiry's resolve_authority).
