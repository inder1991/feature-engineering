# SP-1 Metadata Overlay — Implementation Plan (Overview)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. The **Global Constraints** and the **Shared Contract** below are authoritative; do not redefine shared symbols.

**Goal:** Build the Metadata Overlay (architecture §6 + the [SP-1 design spec](../architecture/2026-06-29-sp1-metadata-overlay-design.md)) on top of the existing `featuregen` package — a platform-owned, event-sourced layer that augments the enterprise catalog with human-confirmed ML facts (availability time, grain, SCD, approved joins, use-case policy), with a merged-view read API, a deterministic profiler, and freshness management.

**Architecture:** A new `overlay_fact` aggregate on SP-0's event store; commands fold state inline (mirroring `run_lifecycle.py`) and append via the existing `append(...)` OCC helper; read models are SP-0 projections; confirmations use SP-0's human-gate; expiry uses SP-0 timers. SP-1 first applies three additive, backward-compatible **SP-0 extensions** (event store, gate model, runtime), then builds the overlay package under `src/featuregen/overlay/`.

**Tech Stack:** Python 3.11 (uv), PostgreSQL, psycopg 3, jsonschema, pytest (+ pytest-postgresql, pytest-timeout). Lint/format: ruff. No new runtime dependencies.

## Global Constraints

- **Python ≥ 3.11**, package `featuregen` under `src/`; tests under `tests/featuregen/`. No new runtime deps.
- **TDD per task:** failing test → watch fail → minimal code → watch pass → commit. Every task ends green.
- **Migrations are two-stage** (`src/featuregen/db/migrations.py::apply_migrations`): Stage-1 Python `MIGRATIONS` list runs first, then Stage-2 `db/migrations/*.sql` in **lexical** order. Editing a Stage-1 `CREATE TABLE IF NOT EXISTS` string does **NOT** alter an existing table — therefore **all SP-1 DDL changes to existing tables (`events`, `human_tasks`, `timers`) are new `.sql` files using `ALTER TABLE ... DROP/ADD CONSTRAINT` / `ADD COLUMN IF NOT EXISTS`**, and must be **idempotent** (guarded) — `tests/featuregen/db/` asserts re-running migrations is a no-op.
- **`.sql` migration ordering (LEXICAL — load-bearing):** files apply in lexical order, so an SP-1 file that ALTERs an existing table MUST sort **after** that table's `.sql`. `timers` is `0502`; therefore SP-1 uses **`0504`–`0507`** (after `0502` timers and `0503`, after `0070` gates, before `0810` privacy). An `008x` prefix would sort BEFORE `0502` and run the timers ALTER before `timers` exists — **do not use it**.
- **Append path:** never INSERT into `events` directly. Use `featuregen.aggregates._append.append(...)` (keyword-only; OCC + provenance + global_seq handled). New streams use `expected_version=0`.
- **Every new event type MUST be registered** (schema_version=1) via the overlay registration hook before any append, or `append_event` raises `SchemaValidationError`.
- **No inline PII** in event payloads or evidence (SP-0 `assert_no_inline_pii` runs on append; evidence stores aggregate metrics only — no raw values, **no raw `MIN`/`MAX`**).
- **Authority model (unchanged):** the engine/profiler *proposes*; deterministic checks validate; a **human** authority *confirms*; the registry governs. Service principals (`actor_kind="service"`) require an attestation; confirmations require a **human** principal.
- **Fail-closed:** `resolve_fact` returns a usable value **only when `status == VERIFIED`**; every other status blocks.

## Key decisions & deviations from the spec

1. **Lifecycle validation is command-level (inline), not the transition engine.** The spec (§2.1 step 3, §3.4) proposed an `overlay_lifecycle_table` + the SP-0 transition engine. Exploration shows the transition engine (`state_machine/transition_table.py`, `engine.py`) is **built but unused in any production path** — `run_lifecycle.py`/`feature_lifecycle.py` enforce lifecycle by **folding state from the event stream and guarding inline**. To stay consistent with the codebase and avoid one-off unused machinery, **overlay command handlers fold state from the stream and validate transitions inline** (a small `_current_overlay_state(stream) -> OverlayState` fold + explicit guards). No `overlay_lifecycle_table`, no `install_transition_table` call. `table_version` stamps the default `1`.
2. **Confirmations are dedicated commands, not `submit_human_signal`.** `propose_fact` opens an SP-0 human-gate **task** (for assignment/eligibility/audit), but the domain transition is performed by dedicated `confirm_fact`/`reject_fact` commands so the **CAS binds to `target_event_id`** (an event id), not the task's `task_version`. The handler resolves fine-grained authority (owner-of-object) and SoD (proposer ≠ confirmer) itself; SP-0 authz policy supplies the coarse capability rows. The task is closed (`cancel_task`/answered) by the same command.
3. **`overlay_fact_id` typed column** is added to `events` (like `request_id`/`feature_id`/`run_id`); `aggregate_id == overlay_fact_id == fact_key` for overlay events, with the consistency CHECK extended explicitly (`request_id/feature_id/run_id IS NULL`).
4. **SP-2 integration:** `confirm_fact` reaching VERIFIED emits nothing extra in SP-1, but the existing `run_lifecycle.py` `waiting_on_fact` park + `FACT_CONFIRMED_RESUME` hook is the documented seam SP-2 will use; SP-1 does **not** build the resume saga.
5. **Expiry is an explicit transactional poller, not a timer handler.** The SP-0 timer runtime only carries `timer_id/kind/task_id` to its queue and builds run-scoped handler contexts — it cannot deliver `fact_key`/`confirmed_event_id` to an overlay handler. So `confirm_fact` schedules an `overlay_expiry` timer (`aggregate="overlay_fact"`, `aggregate_id=fact_key`, `payload={"confirmed_event_id": …}`), and a dedicated **`fire_due_overlay_expiries(conn, *, now)`** poller selects due overlay timers, appends `OVERLAY_FACT_EXPIRED` (CAS on `confirmed_event_id`), and opens the re-verify task. There is **no** `timer.overlay_expiry` HandlerRegistry handler.
6. **`propose_fact` has replacement semantics — no VERIFIED→DRAFT regression.** A new proposal is **denied** if a non-terminal fact already exists for the `fact_key` (current status ∈ {DRAFT, PARTIALLY_CONFIRMED, VERIFIED, REVERIFY, STALE}); a VERIFIED fact stays usable until its own re-verify flow replaces it. Re-proposing is allowed only on an empty stream or after REJECTED (and then only with a different `proposal_fingerprint`). `fold_overlay_state` is defensive — a stray `OVERLAY_FACT_PROPOSED` after a `CONFIRMED` does not regress the folded state.
7. **`approved_join` authority and freshness are side-specific.** Each side (`from_ref`, `to_ref`) resolves its own owner → two tasks; an unknown owner on one side routes **only that side** to the platform-admin/governance queue. Dependency extraction reads the **structured value** (`from_ref` + `to_ref` + `column_pairs`) to index *both* tables and *all* paired columns — never the `display_object_ref` relation string. The `approved_join` value carries `from_ref`, `to_ref`, `column_pairs`, `cardinality` (reconciled on `column_pairs`, not `from_columns`/`to_columns`).
8. **Overlay command registration is idempotent in tests.** `register_command` raises on duplicate and the root test harness resets only the event registry — so `register_overlay_commands()` skips already-registered actions, and the overlay test conftest registers once (session-scoped, after `clear_registry()`).

### Contract pins (review round 2 — resolve cross-file drift; authoritative)
9. **Fact validation error:** `validate_fact_value(fact_type, value, use_case=None)` (use_case **optional**) raises **`FactValidationError`** (defined in `overlay/facts.py`). Every phase imports that name — **not** SP-0's `SchemaValidationError` (which is the event-registry's error).
10. **`schedule_expiry` is introduced in Phase 4, in `overlay/freshness.py`.** Phase 4 creates `overlay/freshness.py` with **only** `schedule_expiry(conn, fact_key, confirmed_event_id, expires_at)`; Phase 7 **extends the same file** with `fire_due_overlay_expiries`/`detect_catalog_changes`/`open_reverify_task`. No forward-phase import.
11. **`proposed_by` is a string** (the actor `subject`) in `OVERLAY_FACT_PROPOSED` — schema, `propose_fact`, profiler, and all test seeds use the string form.
12. **`_OVERLAY_CATALOG` is a `tuple[(action, handler), …]`** (mirrors SP-0's `_CATALOG`); `register_overlay_commands()` iterates it and skips already-registered actions. Phase 6 appends `("run_profiler", _run_profiler)` to the tuple.
13. **Platform-admin authz rows** are seeded for `confirm_fact`/`reject_fact` (below) so governance-queue confirmations pass `execute_command`; the handler still enforces fine-grained authority.
14. **`write_evidence(..., created_by: Mapping)`** — callers pass `identity_to_jsonb(actor)` (a dict), never a raw `IdentityEnvelope`.
15. **`FixtureCatalog(catalog_source=...)` + `add_object(...)`/`set_owner(...)`** is the only API — no `objects=`/`owners=` constructor kwargs. Phase 7 seeds use it.
16. **Column rename identity:** `PostgresCatalog` sets a column's `native_oid` to **`f"{table_oid}:{attnum}"`** (from `pg_attribute`) so column renames are tracked, not degraded to drop/add (tables use the table `oid`).
17. **`confirm_fact` validates the final value** with `validate_fact_value` before appending `OVERLAY_FACT_CONFIRMED` — including any `args["value"]` override on a REVERIFY/STALE correction.
18. **Dependency rows are refreshed on (re)proposal:** `OverlayProjection.apply` DELETEs `overlay_fact_dependency` rows for the `fact_key` before inserting the new set, so a re-proposal after REJECTED never keeps stale dependencies.
19. **`open_reverify_task` is side-specific for `approved_join`** — it reopens a task **per side** (the same per-side `task_assignees` the initial proposal used), not a single task.

### New tables (DDL authored in Task 1.4 / `0507`)
- `overlay_fact_state` (hot, merged-view read): `fact_key text PRIMARY KEY, object_ref text, fact_type text, use_case text NULL, status text, value jsonb NULL, confirmers jsonb NOT NULL DEFAULT '[]', confirmed_at timestamptz NULL, expires_at timestamptz NULL, prior_value jsonb NULL, confirmed_event_id text NULL, updated_seq bigint NOT NULL`.
- `overlay_proposal` (workflow/task read): `fact_key text PRIMARY KEY, status text, proposed_value jsonb, proposal_fingerprint text, draft_event_id text, target_event_id text NULL, evidence_ref text NULL, partial_confirmers jsonb NOT NULL DEFAULT '[]', prior_value jsonb NULL, object_ref text, fact_type text, use_case text NULL, updated_seq bigint NOT NULL`.
- `overlay_evidence` (immutable, written at propose time — NOT a projection): `evidence_id text PRIMARY KEY, fact_key text, table_snapshot_at timestamptz, row_count bigint, sample_size bigint, profile_version text, thresholds_used jsonb, metric_values jsonb, created_by jsonb, created_at timestamptz NOT NULL DEFAULT now()`.
- `overlay_fact_dependency` (general dependency index, projection-maintained): `fact_key text, ref_object text, PRIMARY KEY (fact_key, ref_object)`; index on `(ref_object)`.
- `overlay_catalog_object` (fingerprint snapshot for change detection): `object_ref text PRIMARY KEY, native_oid text NULL, columns_fingerprint text, type_fingerprint text, last_seen_seq bigint NULL, updated_at timestamptz NOT NULL DEFAULT now()`.

### ID prefixes
`fact_key` is a **sha256 hex digest** (not a `mint_id`). Tasks use `mint_id("task")` (existing). Evidence uses `mint_id("eviu")`. (No new `mint_id` prefix needed for facts — the digest is the id.)
