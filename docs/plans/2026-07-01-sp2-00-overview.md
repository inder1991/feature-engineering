# SP-2 Intake + Clarification + Human Gate #1 — Implementation Plan (Overview)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. The **Global Constraints**, the **Additive SP-0 surface**, and the **Shared Contract — Key signatures** below are AUTHORITATIVE; every phase file references them and must not redefine or drift a shared symbol. When a signature here and a phase file disagree, THIS FILE wins.

**Goal:** Build Layer 1 (Intake & Normalization) and Layer 2 (Contract control + human clarification, Human Gate #1) of the platform — architecture §3:88–106 + the [SP-2 design spec](../architecture/2026-07-01-sp2-intake-clarification-design.md) — as a thin domain layer over the existing `featuregen` package. SP-2 turns a data scientist's plain-English intent into a **Confirmed Feature Contract** (the first executable-eligible artifact) via an **auditable LLM** that *structures/suggests*, deterministic checks that *validate/enforce*, a human who *confirms business meaning*, and the registry that *governs*. SP-2 is the **first sub-project that invokes an LLM** and establishes the platform's permanent **auditable-LLM surface**.

**Architecture:** The Feature Contract lifecycle is SP-2's **own event-sourced `feature_contract` aggregate** — its status is **FOLDED from its event stream** (`fold_feature_contract_state`, mirroring SP-1's `overlay/state.py`) and **validated inline** in each command handler (mirroring `overlay/confirmation_commands.py`), **not** an SP-0 run-state enum (there is none — SP-0 only folds run *terminality*, `run_is_terminal`; `run_workflow_state.current_state` is unwired scaffold; `state_machine/engine.py` is built-but-unused), **not** a document-`Stage` transition, and **not** `run_workflow_state`. So — exactly as SP-1 did with its `overlay` aggregate — SP-2 adds **one new aggregate via an additive event-store aggregate-CHECK widening** (mirroring SP-1's `0504_overlay_events.sql`). It rides SP-0's existing **`run` aggregate** for the run **terminal** outcomes (`reject`/`withdraw`/`park`), its **staged-document DAG** (Draft / Assumption-Ledger / Confirmed as frozen content-hashed documents the contract *emits* as write-once `Stage` artifacts), the **`CLARIFICATION`** human-gate, the document **`PRIMARY_SELECTED`** candidate-promotion primitive, the durable runtime, identity/authz + structural SoD, and the security-audit stream. SP-2's other additions are **additive registrations + handlers** — event-types, three document content-schemas + upcasters, an SP-2-owned append-only `llm_call` record store, one backward-compatible human-gate/park-reason migration, one additive service authz row, and a request-owner guard SP-0 does not provide. The new package is `src/featuregen/intake/`.

**Tech Stack:** Python 3.11 (uv), PostgreSQL, psycopg 3, jsonschema, pytest (+ pytest-postgresql, pytest-timeout). Lint/format: ruff. The **LLM adapter is config-gated**: `FakeLLM` is the deterministic CI default (no network); the real Anthropic-SDK Claude adapter (`anthropic`, default model `claude-opus-4-8`, adaptive thinking, structured outputs) is shipped but **never required in CI**. No other new runtime dependencies.

## Global Constraints

- **Python ≥ 3.11**, package `featuregen` under `src/`; SP-2 code lives in `src/featuregen/intake/`, tests under `tests/featuregen/intake/`. No new *required* runtime deps (the `anthropic` SDK is optional / config-gated, imported only inside the real adapter, never at module import time — CI never imports it).
- **TDD per task:** failing test → watch fail → minimal code → watch pass → commit. Every task ends green. All tests run on **`FakeLLM`** — hermetic, reproducible, no network, required in CI (§15). The real adapter is exercised only by an opt-in, config-gated smoke test never gated in CI.
- **Authority model (verbatim, load-bearing):** **the LLM structures/suggests · the platform validates/enforces · the human confirms · the registry governs.** The LLM (`service:intake-agent`) may propose the structured Draft, field scores, candidate calculations, and clarifying questions; it may **never** decide anything is final, write a Confirmed contract, or see raw data/PII. Deterministic platform code validates the content-schema, runs the banking/prohibited-intent screens, runs Minimum Contract Validation, and records every assumption. A **human requester** confirms meaning / picks a candidate / edits fields at Gate #1. The registry governs from Gate #2 (SP-5) onward.
- **Never executable until CONFIRMED (the hard invariant SP-2 exists to protect):** *no Confirmed Feature Contract → no mapping, no compilation, no execution* (design §3:106). A Draft (`status: NEEDS_CLARIFICATION`) is never executable; a run with a **non-empty `open_fields` can never pass Gate #1** (guard `open_fields_empty`).
- **No PII / no data values to the LLM (three-point boundary):** the LLM sees **only** (a) the intent as **redacted, LLM-safe text** from SP-2's `IntentRedactor` seam, and (b) **catalog metadata** — object/column **names, types, asserted grain**, plus any catalog-*declared* enum/code metadata (never profiled values, rows, samples, extrema, or overlay metrics). Enforced at three points with a clean ownership split — **SP-0 classifies** at ingest (`raw_input_classification`), **SP-2 redacts** (`IntentRedactor`, fail-closed on un-redactable / `unscanned`), **SP-2 guards egress** (pre-send hard-fail on data values / un-redacted PII → security-audit). The registered output-schemas are **structural-only** — no PHI/PII in property names, enums, `const`s, or descriptions (server-compiled + cross-call-cached artifact).
- **Event-sourced audit for every LLM call:** each `LLMClient.call` writes **one immutable record** to the SP-2-owned append-only `llm_call` store and emits an `LLM_CALL_RECORDED` event on the run. The record stores the **redacted input itself** (`redacted_input`, replayable — not hash-only), plus provider/model/prompt+schema versions/generation-settings/redaction-version/input-hash/raw-output/validation-result/repair-attempts/latency/cost. Idempotency reuses a record **only** on a **full-identity** key match (below). The store is classified **sensitive / governance-retained / read-controlled**.
- **Fail-closed everywhere:** a malformed LLM structure is a **doubt, not a value** (bounded repair → fail into clarification); a refusal fails into clarification directly (not repair); truncation / timeout / 429 / transient 5xx → bounded retry; non-retryable 4xx → fail closed (auth failures security-audited); an un-redactable-PII intent fails into the manual path (no payload dispatched); a prohibited intent blocks; an exhausted refinement loop auto-parks; an unavailable/unversioned `BankingDomainCatalog` parks. No outcome is silently swallowed, guessed, or executed on. **No silent production fallback:** an enabled-but-unavailable real adapter fails closed into clarification — it never swaps in `FakeLLM`.
- **`FakeLLM` is the default; the real adapter is never required in CI** (Decision D5). All agent code depends on the **`LLMClient` interface — never on a provider**; the provider is isolated to the adapter.
- **Foundation: SP-0 only, + read-only `BankingDomainCatalog`.** SP-2 **builds on SP-0 only** and **reads** two reference surfaces without building them: SP-1's merged-view read API (`resolve_fact` / `list_objects`) for names/types/grain during normalization *only* (never grounding), and the **SP-0-governed, read-only `BankingDomainCatalog`** for the intake banking-boundary + blocked-class + sensitive-proxy screens *only* (never grounding/execution). Reading reference data does not violate "SP-0 only" (Decision D8, ratified). All policy-aware VERIFIED-fact grounding is SP-3.
- **Append path & registration (SP-0 discipline, unchanged):** never INSERT into `events` directly — use `featuregen.aggregates._append.append(...)` (keyword-only; OCC + provenance + `global_seq`). New streams use `expected_version=0`. **Every new event type MUST be registered** (schema_version=1) before any append or `append_event` raises `SchemaValidationError`. **No inline PII** in event payloads (`assert_no_inline_pii` runs on append; `privacy/classification.py:70`) — sensitive bodies are referenced (`blob_`/`doc_`), never inlined.
- **Migrations are additive & lexically ordered** (`src/featuregen/db/migrations/*.sql`, LEXICAL order). SP-2 ships **two** new `.sql` files, both sorting **after** `0070` and SP-1's `0504`–`0507`, and **before** `0810` (privacy), mirroring SP-1's `0504`+`0505` pair: **`0508_feature_contract_events.sql`** — the **event-store aggregate-CHECK widening** admitting the new **`feature_contract`** aggregate (widen `events_aggregate_check` to include `'feature_contract'`, add the typed mirror-id column, rebuild `events_aggregate_id_consistent` with a `feature_contract` branch, add a partial index — the exact recipe of SP-1's `0504_overlay_events.sql`); and **`0509_use_case_onboarding_gates.sql`** — the **gate/park CHECK widening** (mirroring SP-1's `0505_overlay_gates.sql`). Both use `ALTER TABLE ... DROP/ADD CONSTRAINT`, are idempotent (guarded), and **add only allowed values, rewriting no existing row** (additive/backward-compatible — like `0504`/`0505`).

## The additive SP-0 surface SP-2 registers (§2.1 — all backward-compatible)

All six are additive; **no existing SP-0 row, document, event, or aggregate is rewritten** — the two CHECK widenings only *add* allowed values. (Precedent: SP-1's additive `0504_overlay_events.sql` aggregate-CHECK + `0505_overlay_gates.sql` gate-CHECK widenings.)

1. **The `feature_contract` aggregate + an additive event-store aggregate-CHECK migration** (`0508_feature_contract_events.sql`, mirroring SP-1's `0504_overlay_events.sql`) — SP-2's Feature Contract lifecycle is its **own event-sourced aggregate**: widen `events_aggregate_check` to include `'feature_contract'`, add the typed mirror-id column, rebuild `events_aggregate_id_consistent` with a `feature_contract` branch, add a partial index. The aggregate's status is **folded** from its stream (`fold_feature_contract_state`, below), never read from a projection. Adds an allowed aggregate value; rewrites no existing row.
2. **Event types** (registered in SP-0's event-type registry, schema-owned by SP-2) — the contract-lifecycle events emitted on the **`feature_contract`** aggregate: `INTENT_SUBMITTED`, `DRAFT_CONTRACT_PRODUCED`, `CONTRACT_CRITIQUED`, `FIELD_AUTO_RESOLVED`, `CLARIFICATION_REQUESTED`, `CLARIFICATION_ANSWERED`, `CONTRACT_REFINED`, `MINIMUM_CONTRACT_VALIDATED`, `CONTRACT_CONFIRMED`, `USE_CASE_ONBOARDING_REQUESTED`, `INTENT_REJECTED` (carries `OUT_OF_SCOPE` | `PROHIBITED_DATA_CLASS` + catalog `version`), and `LLM_CALL_RECORDED`. The **terminal run outcomes** (`RUN_REJECTED`/`RUN_WITHDRAWN`/`RUN_PARKED`) are emitted on SP-0's `run` aggregate via its existing lifecycle commands.
3. **Document content-schemas** for `DRAFT_CONTRACT`, `ASSUMPTION_LEDGER`, `CONFIRMED_CONTRACT` registered in SP-0's document registry (versioned, with total chained reader-upcasters). The **stage enum is NOT extended** — `contracts/documents.py::Stage` already publishes all three. These are the write-once artifact *kinds* the contract **emits** — an artifact taxonomy, **not** the folded lifecycle status (the `DRAFT_CONTRACT`/`CONFIRMED_CONTRACT` name-collision is a Stage-vs-status distinction, spec §4.6).
4. **The `llm_call` immutable record store** — a new SP-2-owned append-only table (an SP-0-style write-once artifact, like SP-1's `overlay_evidence`), referenced by `llm_call_ref`, classified **sensitive / governance-retained / read-controlled**. A *table*, **not** an event aggregate (distinct from #1); needs no aggregate-CHECK change.
5. **One additive `authz_policy` row (rejection authority)** — because SP-0's `reject` is **validator-only** (`authz/policy.py:42`), SP-2 registers `("reject_intent","","intake-agent","service",None)` admitting the platform/service principal to issue the deterministic intake-rejection terminal outcome `reject_intent` (→ SP-0 `RUN_REJECTED`). Plus the SP-2 command-capability rows (below). This **adds** rows and **changes no existing SP-0 row** — the validator-only `reject` (`authz/policy.py:42`) and data-scientist-owned `withdraw` (`authz/policy.py:41`) are untouched; requester-initiated abandonment reuses that existing `withdraw`, not `reject`. **No onboarding-answer authz row is added** — SP-2 only opens the onboarding gate task and parks; the onboarding-answer/resolution workflow is deferred (§14).
6. **One additive human-gate + park-reason migration** (`0509_use_case_onboarding_gates.sql`, mirroring SP-1's `0505_overlay_gates.sql`) — widens SP-0's `human_tasks` gate CHECK (`0070`: `CLARIFICATION`/`DATA_STEWARD`/`COMPLIANCE`/`INDEPENDENT_VALIDATION`/`FINAL_APPROVAL`) with a new **`USE_CASE_ONBOARDING`** gate value, and registers the **`NEEDS_USE_CASE_ONBOARDING`** park hold-state as an additive `RUN_PARKED` park-reason (base `RUN_PARKED` payload carries only `owner`/`waiting_on_fact`, `run_lifecycle.py`). A gate CHECK widening — distinct from the event-store aggregate-CHECK widening of #1.

**The request-owner guard (SP-2-built; SP-0 does NOT provide it).** SP-0 admits the `data_scientist` role for *any* human (`authz/policy.py`), and `gates/tasks.py::submit_human_signal` checks the responder's **role/scope/quorum** — **never** that the acting `subject` is in the task's `eligible_assignees` (verified: eligibility reads `eligible.get("role"|"scope")`, not the subject). So role-authz alone would let *any* `data_scientist` answer/confirm another author's task. SP-2 therefore adds an explicit guard in its own handlers: `actor_is_request_owner` (at `answer_clarification`) and `confirmer_is_requester_human = actor_is_request_owner ∧ actor_kind=="human"` (at `confirm_contract` / `request_edit` / Gate #1). A mismatch is **denied + written to the security-audit stream**, never counted. In addition, the clarification and Gate #1 tasks SP-2 opens set **`delegation_allowed=False`** (SP-0's `GateTaskSpec.delegation_allowed` **defaults to `True`**, `contracts/envelopes.py:209`) — the subject guard alone is necessary but not sufficient, since a delegate could otherwise stand in.

### The `feature_contract` aggregate — folded status + inline-guard idiom (authoritative — spec §4.6, §11)
The Feature Contract lifecycle is SP-2's **own event-sourced `feature_contract` aggregate** (admitted by the `0508` aggregate-CHECK widening, above). Its status is **FOLDED from the event stream** — never a stored enum, never read from a projection — mirroring SP-1's `overlay/state.py`:

### Feature Contract content-schemas (registered in SP-0's document registry, versioned, upcasted)
- **Closed enum vocabularies** (extended only by a schema-version bump + upcaster): `intake_mode ∈ {definition, hypothesis}` (SP-0 `INTAKE_MODES`); `raw_input_classification ∈ {clean, contains_pii, unscanned}` (SP-0 `RAW_INPUT_CLASSIFICATIONS`); Draft `status = "NEEDS_CLARIFICATION"`; Confirmed `status = "CONFIRMED"`; `observation_intent.kind ∈ {point_in_time, as_of_event}`; `calculation_method.chosen.kind / considered[].kind ∈ {rolling_aggregate, point_snapshot, ratio, distribution_divergence}`; `field_scores.*.source / assumptions[].source ∈ {llm, default, catalog}`; `open_questions[].routed_to ∈ {human, auto}`.
- **`UNKNOWN` representation:** an unresolved semantic field carries the string sentinel `"UNKNOWN"` (SP-0 `documents/draft.py:10`) **and MUST be listed in `open_fields`** (`validate_draft` enforces). A **Confirmed** contract carries no `UNKNOWN` and an **empty** `open_fields`.
- **`DRAFT_CONTRACT` (`schema_version:1`)** — `required: [request_id, intake_mode, raw_input_ref, raw_input_classification, proposed_feature_name, feature_semantics, field_scores, open_fields, assumption_ledger_ref, provenance, status]`; `not: {required: ["raw_input"]}`; `additionalProperties: true` (SP-0 wraps the envelope). `feature_semantics` required: `[entity, entity_grain, observation_intent, calculation_method, windows, filters]` (+ optional `target_definition`). In the **Draft**, `calculation_method` is a **string label** (faithful method name or `"UNKNOWN"`). `field_scores.*` = `{ambiguity, confidence, source}`. `open_questions[]` = `{field, question, blocks_progress, routed_to, ...}`.
- **`CONFIRMED_CONTRACT` (`schema_version:1`)** — `required: [feature_name, intake_mode, raw_input_ref, raw_input_classification, entity, entity_key, feature_grain, observation_intent, calculation_method, assumption_ledger_ref, requires_independent_validation, confirmation, provenance, status]`; `additionalProperties: true`. **Draft→Confirmed renames** (deterministic, applied by the confirm step): `entity_grain` → `feature_grain` (+ derived `entity_key`); `proposed_feature_name` → `feature_name`. The Draft string `calculation_method` (+ `windows`/`filters`) is **reshaped** into the tagged structure below (a shape change, not a rename).
- **Tagged `calculation_method` (`CONFIRMED`, discriminated on `chosen.kind`)** — `{ method_version: int, chosen: method_variant, considered: [method_variant] }`; `method_variant` is a `oneOf` `kind`-discriminated union: `rolling_aggregate {aggregation, window, filter?}` · `point_snapshot {field, filter?}` · `ratio {numerator, denominator, window?}` · `distribution_divergence {measure, window, baseline_window}`; `filter = {concept?, predicate?}`. **SP-3 switches on `chosen.kind` deterministically.**
- **`ASSUMPTION_LEDGER` (`schema_version:1`)** — extends SP-0's registered schema: top-level array is **`assumptions`** (the name SP-0's `ASSUMPTION_LEDGER_JSON_SCHEMA` requires, `documents/draft.py:47` — **not** `entries`); each item `required: [field, value, rationale]` (SP-0), + SP-2 semantic extras `{source ∈ {default,catalog,llm}, ambiguity, confidence, auto_resolved_at}`. - **`confirmation` record (persisted at Gate #1)** = `{confirmed_by, confirmed_at, selected_candidate, rejected_candidates, human_edits[], ambiguity_notes}` (+ confirmer identity/role-claims/`source_of_authority`); rejected sibling `doc_id`s live **only here** (documents are write-once — no per-doc reject event).
- **Upcasters:** each schema is versioned; every bump ships a **total, chained, pure reader-upcaster** registered in SP-0's document registry (read-time projection, never a rewrite — DAG history + content-hashes preserved).

## Appendix — the two running examples (end-to-end, from §Appendix B)

**Definition — `declined_card_auth_count_90d`.** `submit_intent("90-day rolling count of declined card authorizations per customer", mode=definition)` → Draft (entity `customer`; grain `customer × as_of_date` [ledger: default]; method `rolling_count` [ledger]; window `90d` [ledger]; filter `declined authorization` → **open, must-ask**, amb 0.80). One `CLARIFICATION` task to the requester → `auth_result = 'D'`. Re-normalize → all fields resolved → MCV passes → prohibited-intent screen clear → Gate #1: requester confirms the faithful translation → **Confirmed Contract**, `requires_independent_validation=false` → SP-3.

**Hypothesis — abrupt spending-category shift → credit risk.** `submit_intent("customers who abruptly shift spending category are higher credit risk", mode=hypothesis)` → Draft with `target="higher credit risk"` (policy-sensitive, must-ask) + calc-method must-ask. `StubCandidateGenerator` makes **one** LLM call → 3 candidate documents (distinct-MCC delta; top-category-share drift; distribution divergence), each with a rationale + cheap `signals` (no IV/WoE). Risk-flag screen fires → `requires_independent_validation=true`. Prohibited-intent screen: target pinned to a permitted, non-blocked label (a `sensitive_proxy_hints` match → clarification/compliance review; a `blocked_data_classes` protected attribute would **block** as `PROHIBITED_DATA_CLASS`). At Gate #1 the requester picks one candidate (document `PRIMARY_SELECTED`; losers recorded in the confirmation record only), confirms the pinned target → **Confirmed Contract** (`intake_mode=hypothesis`, `selected_candidate`/`rejected_candidates` recorded, `requires_independent_validation=true`) → SP-3 (and the flag drives Gate #2 at SP-5).
