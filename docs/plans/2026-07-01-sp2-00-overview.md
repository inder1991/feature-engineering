- **Foundation: SP-0 only, + read-only `BankingDomainCatalog`.** SP-2 **builds on SP-0 only** and **reads** two reference surfaces without building them: SP-1's merged-view read API (`resolve_fact` / `list_objects`) for names/types/grain during normalization *only* (never grounding), and the **SP-0-governed, read-only `BankingDomainCatalog`** for the intake banking-boundary + blocked-class + sensitive-proxy screens *only* (never grounding/execution). ### The `feature_contract` aggregate â€” folded status + inline-guard idiom (authoritative â€” spec Â§4.6, Â§11)
The Feature Contract lifecycle is SP-2's **own event-sourced `feature_contract` aggregate** (admitted by the `0508` aggregate-CHECK widening, above). Its status is **FOLDED from the event stream** â€” never a stored enum, never read from a projection â€” mirroring SP-1's `overlay/state.py`:

- **Inline-guard idiom (mirrors `overlay/confirmation_commands.py`):** every SP-2 command handler **folds the current status first**, then **validates the advance inline before appending** â€” `state = fold_feature_contract_state(stream); if state.status not in <allowed_from>: return DENY(...)` â€” including a **no-regression guard** (a fold already at/past a terminal/confirmed status refuses a conflicting re-advance). It is **NOT** the declarative `state_machine/engine.py` (built-but-unused â€” SP-1 deliberately did not wire it) and **NOT** the `run_workflow_state` projection (unwired scaffold).

### Feature Contract content-schemas (registered in SP-0's document registry, versioned, upcasted)
